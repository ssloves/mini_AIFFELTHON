# 개선 방향 보고서 — 시스템은 그대로, 데이터셋·메트릭으로 모델을 분해 해석한다

> 최종 목표: **모델 아키텍처(agent/)는 고정**하고, **데이터셋(1순위)과 평가 메트릭(2순위)을 고도화**하여
> 우리 GraphRAG를 "어디서·왜 이기고 지는지" 다각도로 분해·해석한다.
> 아키텍처(3순위)는 *정말 크리티컬한 문제가 아니면 수정하지 않는다.*

---

## 0. 핵심 결론 요약 (TL;DR)

1. **지금 느린 게 아니라 메트릭이 모델을 과대/혼합 평가하고 있다.** Context Recall(+0.241)만이 생성과
   무관한 *순수 구조 지표*라 정직하게 귀인 가능하고, Answer Accuracy(+0.356)·Faithfulness(+0.216)는
   **검색 외 요인(생성 프롬프트·재시도 루프)이 섞인 혼합 지표**다(§1.2).
2. **대조군과 실험군의 차이는 "추론 방식뿐"이 아니다.** 모델(gpt-4o)·임베딩·청크 예산은 통제됐지만,
   생성 프롬프트·Verifier 재시도 루프·충돌 게이팅이 **대조군엔 없다**. 따라서 격차를 전부 "그래프 덕분"이라
   말하면 논리적 비약이다. → 이 confound를 **메트릭 축의 ablation으로 분해**하면, 코드를 건드리지 않고도
   "그래프가 순수하게 기여한 양"을 정량화할 수 있다(§3.3).
3. **그래프 경로 로그는 절반만 있다.** 엣지 *타입별 카운트*(`expanded_via`)와 청크별 *도착 라벨*(`via`)은
   있으나, "어느 노드에서 어느 엣지를 타고 어느 노드로 갔는지"의 **경로(triple)** 는 기록되지 않는다
   (특히 개념 브리지·참조 홉은 출발/중간 노드가 유실). 이는 **관찰가능성 보강**으로 해결되며 검색 로직·
   점수에는 영향이 없다(§1.1, §3.6).
4. **표본이 부족해 결론 자체가 불가능한 영역이 있다.** hop3가 2문항뿐이라 "멀티홉 우위"는 통계적으로
   말할 수 없다. → 데이터셋 확장이 1순위인 이유(§2).
5. **3순위(아키텍처)는 동결.** 피드백의 Router few-shot(11번)·Reasoner 충돌억제(13번)는 곧바로 고치지
   않고, **새 메트릭이 "그게 지배적·고립된 실패 원인"임을 입증할 때만** 적용하는 *조건부 예비안*으로 둔다(§4).

## 1. 현 상태 점검 (코드·실험 로그 실측 근거)

### 1.1 그래프 경로 로깅 현황 — "카운트는 있고, 경로는 없다"

현재 상태 추적(`agent/graph_state.py`의 trace, `agent/nodes.py`의 retriever)이 남기는 것:

- `expanded_via`: `{엣지타입: 개수}` — 예) `concept_conflict_anchor: 1`, `CONFLICTS_WITH: 2`,
  `DEFINES(concept-bridge): 6`, `REFERENCES: 12`. → **얼마나 확장했는지(양)** 는 알 수 있다.
- 각 컨텍스트 청크의 `via` 라벨: `seed` / `concept-conflict` / `conflict` / `concept:상시근로자` /
  `reference`. → **그 청크가 어떤 경로로 들어왔는지(종류)** 는 안다.
- `conflicts`: `{src, id, axis, note, verified}` — CONFLICTS_WITH는 **출발→도착 + 축**까지 있다.

빠진 것(= 사용자가 원하는 로그):

- **개념 브리지**(`_concept_bridge`)는 Cypher가 도착 청크 `o`와 다리 개념 `k.name`은 돌려주지만
  **출발 청크 `c`를 RETURN하지 않는다** → "어느 청크가 어떤 Concept를 거쳐 어디로 갔는지" 경로가 끊김.
- **참조 홉**(`_references`)은 `c-[:REFERENCES]->(a:Article)-[:CONTAINS]->(o)` 경로인데 도착 `o`만
  돌려주고 **출발 청크·중간 Article 노드를 버린다**.
- 결과적으로 "node → edge → (intermediate) → node" **트리플 단위 경로 로그**가 없다.

**판정**: 이건 *모델의 추론을 바꾸는 일이 아니라 관찰가능성(observability) 보강*이다. Cypher의 RETURN에
출발/중간 노드를 추가하고 trace에 트리플을 적재하면 되며, **검색 점수·재랭킹·최종 컨텍스트는 불변**이다.
→ 3순위(아키텍처) 동결 원칙을 깨지 않는다. 구체안은 §3.6.

### 1.2 대조군 공정성 점검 — "차이가 추론 방식뿐인가?" → **아니다**

| 요인 | 실험군(GraphRAG) | 대조군(강화 Naive) | 통제? |
|------|------------------|--------------------|:----:|
| 생성 LLM | gpt-4o | gpt-4o | ✅ 통제됨 |
| 임베딩/검색 substrate | text-embedding-3-small + Neo4j 벡터 | 동일 | ✅ 통제됨 |
| 청크 예산(개수) | 최종 컨텍스트 수 | 동일하게 맞춤(min 8) | ✅ 통제됨 |
| **검색 방식** | 벡터 + 개념앵커/브리지/참조 | 벡터만(조문 다양화) | ◎ *의도된 변인* |
| **생성 프롬프트** | REASONER_SYS(충돌표면화·인용복사·수치엄수·구조화 JSON) | NAIVE_SYS(단순 "발췌만, 없으면 모름") | ❌ **confound** |
| **자기수정 루프** | Verifier + 재시도 ≤2(폭 확대 재검색) | 없음(단발) | ❌ **confound** |
| **충돌 게이팅/flag** | 결정론적 게이팅 + flag 산출 | 없음 | ❌ **confound** |

**함의(매우 중요):**
- **Context Recall은 생성 이전 단계의 지표**라 위 confound가 끼어들 여지가 없다. 따라서 **+0.241은
  그래프 검색의 순수 구조 기여**로 정직하게 귀인할 수 있다.
- 반면 **Answer Accuracy(+0.356)·Faithfulness(+0.216)는 "더 좋은 검색 + 더 좋은 프롬프트 + 재시도"의
  합작**이다. 이 셋을 분리하지 않으면 "그래프가 정확도를 0.356 올렸다"는 **과대 귀인**이 된다.
- 즉 대조군은 *검색 예산*은 공정하게 맞췄지만 *생성 환경*은 불공정하게 빈약하다. 이 비대칭을 인정하고,
  메트릭 축에서 **요인을 하나씩 더한 사다리(ablation ladder)** 로 분해하는 것이 정답이다(§3.3).
  핵심은 이 분해가 **모델 코드를 전혀 바꾸지 않고**, 기존 함수(`reasoner`, `verifier`, `baseline_context`,
  `retrieve`)를 *평가 하니스에서 조합만 달리해* 가능하다는 점이다.

### 1.3 채점 정밀화의 전제 — 청크 구조 실측 (피드백 #1 "먼저 1줄 찍기"에 대한 답)

`data/chunks/all_chunks.jsonl` 실측 결과, 청크는 다음 필드를 가진다:
`chunk_id, source_doc, doc_type, chapter, article, article_title, paragraph, hierarchy_path, page,
section, context_header, text, token_count, references, keywords, defines`.

- `article`(조), `page`, `section` 필드는 **존재**한다. `paragraph` 필드도 있으나 **대부분 null**이다.
- 이유: 청크 단위가 **대체로 "조" 단위**다. 항(項)은 큰 조가 분할될 때만 chunk_id 접미사로 표현된다
  (예: `근기령_제7조의2_제1-2항`). 지침은 조 구조가 없어 **`지침_p53_s119`처럼 page/section 앵커**를 쓴다.

**판정(피드백 #1 현실 점검):**
- "조→항/호 정밀화"는 **청크가 항 단위로 분할된 경우에만** 분해능이 생긴다. 조 단위 청크에서는 항-gold가
  결국 같은 청크로 매핑되므로 분해능이 없다 — 이 한계를 보고서·코드에 *명시*해야 헛수고를 막는다.
- **6번 가짜 hit의 진짜 원인은 항이 아니라 "지침 문서단위(`*`) 매칭"** 이다. 지침 청크는 `p/s` 앵커가
  있으므로, gold를 page/section으로 라벨링하면 정밀 매칭이 되어 **6번의 Recall 1.0이 정당하게 내려간다**.
  → 즉 피드백 #1의 실효는 "항/호"보다 **"지침 `*` → page/section 정밀화"** 에서 나온다.

## 2. 1순위 — 데이터셋 고도화 (모델을 분해하는 "측정 자(尺)"를 다시 만든다)

데이터셋은 모델을 비추는 거울이다. 거울이 18칸·hop3 2칸뿐이면 모델의 진짜 능력/한계가 안 보인다.
목표는 단순 증량이 아니라 **"어느 축에서 이기고 지는지 읽히도록 설계된" 데이터셋**이다.

### 2.1 문제 (exp_report 한계 직결)
- **표본 부족·불균형**: 18문항, hop1=4·hop2=12·**hop3=2**. → "멀티홉 우위" 결론 불가(보고서 H2 잠정).
- **라벨 혼재**: `type=conflict` 12개 중 5개(4·5·6·9·12)는 단일 문서라 "양쪽 기준" 정의 불성립.
- **by-construction(순환) 우위**: 충돌 문항의 gold가 곧 우리가 손으로 큐레이션한 7개 CONFLICTS_WITH
  쌍과 일치 → 그래프가 "정답을 미리 주입"하는 구조라, 충돌 Recall 우위의 일부는 *성능이 아니라 설계*다.
- **gold 입도(粒度) 거침**: 조 단위 + 지침 문서단위 → 6번처럼 *틀린 청크인데 hit*로 집계.
- **음성 표본 빈약**: distractor 6개 중 "이름은 같지만 결과는 같아 충돌 아님"(13번류)이 사실상 1~2개뿐 →
  과잉탐지를 측정할 표본이 너무 적다.
- **앵커 밖 일반화 미측정**: hop3 연쇄추징의 핵심 조문(조특법 제29조의8)은 7개 앵커 밖이라 그래프가
  절반만 도달(3·7번). 그런데 이런 "앵커 밖" 문항이 거의 없어 *일반화 실패가 가려진다*.

### 2.2 개선 방향 (각 항목이 "모델의 어떤 축을 분해"하는지 명시)

1. **규모 확장: 18 → 60~100+ 문항.** 단, 무작정 늘리지 말고 아래 설계 매트릭스의 *칸(cell)* 을 채우는
   방식으로 늘린다. 특히 **hop3를 최소 10문항 이상** 확보해야 멀티홉 주장에 통계적 힘이 생긴다.
   - *분해하는 축*: hop 심화에 따른 Recall 감쇠 곡선(그래프 vs Naive)을 처음으로 *통계적으로* 그릴 수 있다.

2. **층화 설계 매트릭스.** 각 문항에 (hop ∈ {1,2,3}) × (유형 ∈ {문서간 충돌, 단일문서 정밀, 동명-무충돌
   distractor, 무관 distractor, 무응답(정답=문서에 없음)}) × (연결 기제 ∈ {REFERENCES, Concept/CONFLICTS_WITH})
   라벨을 단다.
   - *분해하는 축*: "그래프가 **참조 기반 홉**에서 강한가, **개념 기반 홉**에서 강한가"를 칸별로 읽는다.
     (EDA에서 cross 참조가 조특법↔조특령에 편중됐다는 사실의 *검증 데이터*가 된다.)

3. **순환성 차단용 held-out 충돌.** gold가 **7개 앵커 쌍 밖**의 조문을 요구하는 충돌/연쇄 문항을 다수 추가
   (제29조의8 연쇄류). 손으로 앵커한 충돌만 잘 푸는지, *처음 보는 관계도* 그래프로 일반화하는지 가른다.
   - *분해하는 축*: "암기(앵커) vs 일반화"의 분리. 그래프의 진짜 구조적 가치가 여기서 드러난다.

4. **정밀 gold 라벨(항/호 + 지침 page/section).** 법령은 조-항(-호)까지, 지침은 page/section 앵커까지
   라벨링한다(§1.3대로 항은 분할 청크에서만 분해능). → 메트릭 축의 "가짜 hit 제거"(§3.1)와 짝을 이룬다.
   - *분해하는 축*: Recall의 *진짜값*. 6번 같은 "찾은 척" 사례를 데이터 라벨 단계에서 솎아낸다.

5. **적대적/음성 표본 보강.** (a) **동명-무충돌**: 이름은 같으나 *결과가 같아* 충돌이 아닌 문항(13번류)을
   여러 개. (b) **용도상이**: 같은 "단시간"이라도 고보령 제3조①(피보험 자격)과 제12조⑤(우선지원 규모)처럼
   *목적이 다른* 쌍을 충돌로 오인하는지. (c) **무응답**: 어떤 문서로도 답할 수 없어 "모름"이 정답인 문항.
   - *분해하는 축*: 과잉탐지(거짓 충돌)와 과소응답(헛걸음)을 정량화 — 신뢰성의 음성 측정.

6. **인용(provenance) gold.** 각 문항에 *기대 인용 청크 집합*을 라벨링한다.
   - *분해하는 축*: 답 텍스트가 아니라 **근거 추적의 정밀/재현율**을 잴 수 있다(그래프의 핵심 셀링포인트).

7. **질문 패러프레이즈(구어 변형) 세트.** 같은 취지를 "주 14시간 파트타임"처럼 구어로 바꾼 변형을 둔다
   (14번이 정확히 이 어휘 간극에서 양 시스템 모두 실패).
   - *분해하는 축*: 실패가 *검색-어휘* 문제인지 *추론* 문제인지 분리. 라우팅/임베딩 강건성 측정.

### 2.3 규모 확장의 현실적 방법과 함정
- **KG에서 반자동 생성**: CONFLICTS_WITH 쌍 → 충돌형 질문, REFERENCES 사슬 → 멀티홉 질문, 단일 조 →
  단순 질문, 동명개념(용도상이) → distractor를 *템플릿*으로 뽑고 **전문가가 검수**한다. KG에서 뽑으면
  gold 청크를 *이미 알고* 있어 라벨이 정확하다.
- **함정(정직 고지)**: KG에서 생성한 문항은 §2.1의 by-construction 순환을 *재생산*한다. 따라서 생성형
  문항과 **사람이 손으로 만든 held-out(앵커 밖)** 문항을 반드시 섞고, 보고 시 두 군을 분리 집계한다.

8. **Gold Path 라벨(추가).** gold answer·gold chunk에 더해, 질문 진입점에서 gold 근거에 이르는
   **필수 관계 사슬**을 라벨링한다 — `(출발노드, 엣지타입, [중간노드], 도착노드)` 트리플의 집합 또는
   *허용 경로 부분그래프*(등결성 대비 OR-그룹 포함). 노드=청크/조문/개념, 엣지=REFERENCES/DEFINES/
   CONFLICTS_WITH/DELEGATES 등.
   - *분해하는 축*: "정답을 맞혔나"를 넘어 **"법리적으로 옳은 경로로 도달했나"**(§3.8 메트릭의 정답지).
   - *전제*: §3.6의 trace 트리플 로깅이 있어야 모델 경로와 대조 가능.

### 2.4 데이터셋 무결성 — "규칙기반인가? 환각 없는가?" 검증

**생성 방식**: KG 템플릿(규칙) 기반이 핵심이고, LLM은 *표현 다듬기(패러프레이즈)* 에만 제한적으로 쓴다.
KG에서 실제 엣지/노드를 샘플링해 만들면 gold answer·chunk·path가 *샘플링한 KG 요소 그 자체*라
**라벨 환각 위험은 사실상 0**(provenance-exact)이다.

**그러나 무결성은 "생성 방식"이 아니라 "검증 게이트"로 확보된다.** 정직한 두 한계:
- **데이터셋 무결성 ⊆ KG 무결성**: KG의 REFERENCES는 정규식 추출이라 noise가 있다(감사로 수치화는 했으나
  100%는 아님). 그 위에 만든 멀티홉 문항은 KG 오류를 *상속*한다. (CONFLICTS_WITH는 전문가 큐레이션이라 깨끗.)
- **LLM 개입 시 환각 가능**: 자연스러운 질문/distractor 생성에 LLM을 쓰면 없는 조항·틀린 수치·가짜 충돌을
  지어낼 수 있다 → 반드시 *제공 청크에 근거 제한* + 검증.

**무결성 게이트(전 문항 필수 통과)**:
- **G1 Provenance**: 모든 gold chunk_id·path edge가 KG에 *실존*하는지 자동 검사(없으면 폐기).
- **G2 수치 Grounding**: gold_answer의 모든 수치·기간·연령·인원이 gold 청크 text에 *verbatim* 존재하는지.
- **G3 응답가능성·유일성**: 최소 1개 gold 청크가 답을 지지하는가; distractor는 *충돌을 지지하지 않음* 확인.
- **G4 Held-out 인적분 교차검수**: 앵커 밖 손저작 문항은 원문 대조 + 2인 독립 검수(불일치는 폐기/수정).
- **G5 누출·중복 차단**: 기존 18문항 및 KG의 학습 요소와 중복/누출되지 않도록 검사.

→ 따라서 보고 시에는 **"생성만으로 무결"이 아니라 "게이트 통과로 무결에 수렴"** 이라 표기하고, KG 파생
   문항에는 **참조감사 신뢰구간을 문항 신뢰도로 함께 표기**한다.

## 3. 2순위 — 평가 메트릭 고도화 (모델을 "건드리지 않고" 분해하는 도구)

원칙: 모든 메트릭 개선은 **측정 장치(eval 하니스)에서만** 이뤄진다. 기존 함수를 *조합·채점만 달리* 하며
agent/ 코드는 불변이다. 사용자의 피드백 #1·#2가 여기에 직접 들어온다.

### 3.1 조→항/호 + 지침 page/section 정밀화, 6번 가짜 hit 제거 (피드백 #1)
- **무엇을**: 채점 매칭 단위를 (문서, 조)에서 (문서, 조, 항범위) 로 좁히고, 지침은 (문서, page/section)로
  바꾼다. chunk_id 접미사(`제1-2항`, `p53_s119`)를 파싱해 gold 항범위/페이지와 교집합을 본다
  (참조감사에서 쓴 항범위 파서를 재사용).
- **왜**: 6번은 "지침 아무 청크나 = hit"라 Recall 1.0이지만 Accuracy 0이었다. 페이지/섹션 매칭이면
  *정작 필요한 인건비 차감 규정*이 없었음을 드러내 Recall이 정당히 내려간다.
- **실행 순서(피드백 그대로)**: ① 청크 필드 1줄 확인(완료: §1.3 — `paragraph`는 자주 null, `page/section`
  존재) → ② `parse_gold/norm_para/covers/recall` 4개를 항범위·page/section 인지로 교체 → ③ **6번만
  먼저 재실행**해 Recall이 1.0에서 내려가는지 확인(내려가면 가짜 hit 포착 성공) → ④ 전체 재실행.
- **정직한 한계**: 조 단위 청크에서는 항 분해능이 없다(§1.3). 그래서 이 정밀화는 *지침·항분할 청크*에서만
  효과가 있고, 그 사실을 보고서에 명시해 "정밀화했으니 Recall은 완벽"이라는 오해를 막는다.

### 3.2 judge에 사실성 체크 추가 / 17번 오염 제거 (피드백 #2)
- **문제**: 현재 Faithfulness judge는 "컨텍스트와의 정합"만 본다. 그래서 17번처럼 답변이 *미확인 도메인
  사실*("군필 39세" 등 verified=false 세트3)을 담아도 1.0을 줬다. judge가 **진위**를 못 본다.
- **개선**:
  1. **인용-귀속 사실성**: 답변의 각 수치/요건이 *실제로 인용된 청크 text 안에* 근거 문장이 있는지
     문자열·수치 단위로 확인(LLM이 아니라 결정론적 대조 우선). 근거 없는 단정은 감점.
  2. **오염 블록리스트**: 프로젝트가 verified=false로 분류한 주장(예: 군필 39세, 세트3)을 *알려진 오염
     목록*으로 두고, 답변이 이를 포함하면 사실성 점수에서 명시적 페널티. → 17번이 비로소 감점된다.
  3. **모수 지식 누출 차단**: Accuracy judge가 "그럴듯함"이 아니라 *gold + 출처 대조*로만 채점하도록
     루브릭을 고정(생성모델의 사전지식으로 맞힌 11번류를 "근거 없는 정답"으로 별도 표시).
- *분해하는 축*: "맞는 말"과 "근거 있는 맞는 말"을 분리. 그래프의 가치는 후자에 있으므로 정직해진다.

### 3.3 confound 분해 — ablation 사다리 (피드백 #3의 "정말 구조적 이점인가?"에 대한 실행 답)
모델을 바꾸지 않고, **기존 부품을 평가 하니스에서 조합**해 요인을 하나씩 더한 사다리를 만든다:

| 단계 | 구성(전부 기존 함수 재사용) | 직전 대비 추가된 요인 |
|------|------------------------------|------------------------|
| **B0** | Naive(현행): 벡터검색 + 단순 프롬프트, 단발 | (기준선) |
| **B1** | B0의 컨텍스트 + **Reasoner 프롬프트로 생성** | 생성 프롬프트 품질 |
| **B2** | B1 + **Verifier·재시도 루프** | 자기수정 루프 |
| **B3** | **그래프 검색(앵커 OFF)** + Reasoner + 루프 | 그래프 확장(앵커 제외) |
| **B4** | 그래프 검색(앵커 ON) = 현행 GraphRAG 전체 | 개념 충돌 앵커 |

귀인(데이터로 분해):
- **검색 순수 기여** = Recall(B3/B4) − Recall(B0) → *생성과 무관*하므로 가장 깨끗(이미 +0.241이 이 신호).
- **프롬프트 기여** = Acc(B1) − Acc(B0).
- **재시도 기여** = Acc(B2) − Acc(B1).
- **그래프(앵커 제외) 기여** = 지표(B3) − 지표(B2).
- **충돌 앵커 기여** = 지표(B4) − 지표(B3).

이 표 하나로 "Accuracy +0.356이 그래프 덕인가, 프롬프트 덕인가"를 *수치로* 가른다.
(실행상 B1=`baseline_context`+`reasoner` 호출, B2=거기에 `verifier`+루프, B3=`retrieve`를 simple 파라미터로
호출 — 모두 기존 함수 조합이라 **아키텍처 무수정**.)

### 3.4 `via` 기반 "그래프 한계기여" 귀인
컨텍스트 청크마다 이미 `via`(seed/concept-conflict/concept/reference) 라벨이 있다. 이를 이용해 문항별로
**"gold 청크가 벡터 seed만으로도 잡혔는가, 아니면 그래프 전용 경로로만 잡혔는가"** 를 집계한다.
- gold가 seed로도 잡혔다면 → 그 문항에서 그래프의 한계기여 = 0.
- gold가 concept/conflict/reference로만 잡혔다면 → 그래프가 *없었으면 놓쳤을* 근거 = 순수 기여.
- *분해하는 축*: "그래프가 실제로 차이를 만든 문항 비율"을 한 줄 숫자로. 추가 LLM 호출 없이 가능.

### 3.5 충돌 판정의 혼동행렬(P/R/F1)과 과소응답률
- 라벨된 충돌/무충돌(동명·용도상이·무관) 세트 위에서 `conflict_flagged`(또는 답변 충돌서술)에 대한
  **정밀도·재현율·F1**을 보고한다. 단순 "오탐률 0.5"보다 모델의 신뢰성을 풍부하게 분해한다.
- **과소응답률**: "모름/근거없음"으로 회피한 비율을 시스템별로(특히 Naive가 1·5·9에서 회피). Recall↔회피의
  교환관계를 드러낸다.

### 3.6 그래프 경로(node→edge→node) 로깅 + 경로정합성 메트릭 (사용자 요청 직답)
- **로깅(관찰가능성, 점수 불변)**: `_concept_bridge`/`_references` Cypher RETURN에 출발 청크·중간 노드
  (Concept.name / Article)를 추가하고, trace에 `{src, edge, mid, dst, concept/axis}` 트리플 리스트를 적재.
  CONFLICTS_WITH는 이미 src/axis가 있으니 그대로 트리플화. → "어느 노드에서 어느 엣지로 어디로" 완성.
- **경로정합성 메트릭(신규 해석 지표)**: gold 청크에 *도달한 경로*가 법리적으로 타당한 관계인지 채점
  (예: 충돌 문항이라면 앵커 쌍의 axis가 질문 개념과 일치하는가). trace를 *증거*에서 *평가 신호*로 승격.
- *분해하는 축*: "정답을 맞혔다"를 넘어 **"정답에 옳은 길로 도달했다"** 를 본다 — GraphRAG의 본질적 강점.

### 3.7 분산·재현성
- 현재 1회 실행. 라우팅·judge가 확률적이므로(11·13·16·17이 route에 민감) **다중 실행/패러프레이즈로
  평균±신뢰구간**을 보고하고, route 변동 빈도를 함께 남긴다.

### 3.8 Gold Path 기반 추론경로 메트릭 (신규 — 논리적 유효성 검증 완료)

§2.2-8의 gold path를 정답지로, 모델의 실제 trace(§3.6) 경로와 대조해 **"옳은 관계로 도달했는가"** 를 잰다.
§3.6의 경로정합성보다 강한 버전 — 규칙 기반 자동판정이 아니라 *법문에서 독립 저작된 gold path* 와 비교한다.

**왜 자의적이지 않은가 (유효성 근거).** 우리 엣지는 실제 법적 관계라, 질문 진입점→gold 근거의 *필수 관계
사슬*은 법령 구조가 정하는 객관적 속성이다(예: 연쇄추징은 지침→조특법 제29조의8→조특령 제26조의8).
gold answer를 객관적으로 라벨링하듯 gold path도 객관적으로 라벨링할 수 있다.

**유효성을 보장하는 3대 가드레일 (없으면 "끼워맞추기"가 되어 무효).**
1. **독립 저작**: gold path는 *모델 trace가 아니라 법문/스키마에서* 만들고 실행 전 동결. trace 보고 역산하면
   동어반복 → 무효. (gold answer와 동일 규율.)
2. **등결성(equifinality) 허용**: 정답에 이르는 유효 경로는 여럿(참조 경유 vs 개념 경유)일 수 있으므로,
   **정확한 순서 일치가 아니라 "필수 관계 커버리지 + OR-그룹 대체경로"** 로 채점. *시퀀스 편집거리는
   유효한 다른 경로를 처벌하므로 금지.*
3. **보완 지표(대체 아님)**: Context Recall=올바른 *노드* 도달, Gold Path=올바른 *관계*로 도달. 별개 신호다.

**채점식.**
- **필수관계 재현율** = |모델 경로가 실제로 거친 관계 ∩ gold 필수관계| / |gold 필수관계| (OR-그룹은 그룹 내
  하나라도 충족 시 인정).
- **경로 정밀도** = 모델이 gold 근거에 도달하며 *허위/무관 엣지*를 얼마나 적게 썼는가(불법 경유 penalize).
- 두 값을 F-측도로 종합하되, 등결성 때문에 *재현율을 우선* 본다.

**판별타당도 (기존 지표가 못 잡는 신호 — 이미 데이터에 증거).**
- **11번**: Accuracy 1.0이지만 route=simple이라 CONFLICTS_WITH를 안 탔다(정답이나 *모수지식으로 맞힘*).
  Recall·Accuracy는 통과시키나 **Path 메트릭만 "옳은 이유 아님"을 포착** → 거짓 성공을 분리한다.
- **B3(앵커 OFF) vs B4(앵커 ON)**: "gold에 CONFLICTS_WITH로 도달한 비율"로 앵커의 *기제적 기여*를 직접 측정.

**3분할 해석(다각도 분해의 핵심 산출).** 문항을 ① *옳은 경로로 정답* ② *경로 없이/틀린 경로로 정답(요행)*
③ *정답 실패* 로 나눈다. ②의 비율이 곧 "정확도가 부풀려진 정도"이며, GraphRAG가 *추론 시스템*인지
*운 좋은 검색기*인지를 가른다.

**적용 범위·정직한 한계.**
- 그래프 전용 지표다(Naive는 경로가 없어 0 → *비교용 아님, 그래프 내부 해석용*).
- §3.6 trace 트리플 로깅이 *전제*다(브리지·참조의 출발/중간 노드 복원 전에는 측정 불가).
- gold path 저작 비용이 있다 → §2.3 KG 반자동 생성으로 *필수관계를 KG에서 직접 추출*해 비용을 낮추되,
  held-out 문항은 사람이 저작(순환 방지).

## 4. 3순위 — 모델 아키텍처 (원칙적 동결, 조건부 예비안만 보유)

### 4.1 동결 원칙
시스템(agent/)은 **그대로 둔다.** 이유는 단순하다 — 지금 우리가 가진 건 "1회·18문항"이라, *무엇이 진짜
지배적 실패 원인인지 아직 모른다.* 원인을 모른 채 프롬프트를 만지면, 18문항에 다시 과적합할 위험이 크다
(이미 한 번 모든 지표가 1.0으로 과적합됐던 전례). **먼저 데이터·메트릭으로 원인을 분해한 뒤** 결정한다.

### 4.2 피드백 #3·#4는 "조건부 예비안"으로 보류
- **#3 Router few-shot(11번 cross-doc→simple 오분류)**: 곧바로 고치지 않는다. 대신 §2.2의 gold `route`
  라벨로 **Router 정확도를 메트릭화**한다. 만약 (a) 라우팅 오류가 Recall 손실의 *지배적* 원인이고
  (b) 그것이 *고립된* 단일 원인임이 수치로 증명되면 — 그때 *최소한의 few-shot 예시 추가*만 허용한다.
- **#4 Reasoner "결과 같으면 충돌X"(13번 동명-무충돌 과잉탐지)**: 마찬가지. §3.5의 충돌 정밀도와
  §2.2의 동명-무충돌 표본으로 **과잉탐지를 정량화**한다. 정밀도가 명백히 낮고, 원인이 게이팅 로직 단독으로
  좁혀질 때만 *프롬프트 한 줄* 수준의 최소 개입을 검토한다.
- 두 경우 모두 **"메트릭이 먼저 유죄를 입증"** 해야 하며, 그 전까지는 *해석 대상*이지 *수정 대상*이 아니다.

### 4.3 "구조처럼 보이지만 해법은 데이터/메트릭"인 사례
- **by-construction 앵커 일반화 갭**(3·7번): 가장 "구조 문제"처럼 보이지만, 해법은 코드가 아니라
  **§2.2-3 held-out 충돌 데이터**와 **§3.4 via 귀인 메트릭**이다. 아키텍처를 바꾸는 대신, *그래프가
  앵커 밖에서 얼마나 일반화하는지를 측정*해 한계를 정직하게 보고한다.
- **경로 로깅**(§3.6)은 관찰가능성이라 동결 원칙과 무관하게 진행한다(점수·결정 불변).

### 기대 효과 (다각도 해석의 완성)
- **헤드라인**은 confound-free 지표인 **Context Recall**(+ via 귀인)로 — "그래프는 근거를 덜 놓친다"는
  주장을 정직하게 세운다.
- **분해표**(B0~B4)로 Accuracy 격차의 출처를 *검색/프롬프트/재시도/앵커*로 나눠, "구조적 이점"의
  실제 크기를 수치로 제시한다.
- **칸별 결과**(hop×유형×기제)와 **경로정합성**으로, 모델이 *어디서·어떤 엣지로* 이기고 *어디서* 지는지를
  설명한다 — 모델은 그대로 둔 채, 데이터셋과 메트릭만으로 모델을 **다각도로 분해·해석**한다.

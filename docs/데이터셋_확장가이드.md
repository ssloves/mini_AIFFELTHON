# 데이터셋 확장 가이드 (18 → 60~100문항)

> 목표: 단순 증량이 아니라 **"모델이 어느 축에서 이기고 지는지 읽히도록 층화 설계된"** 평가셋.
> 원칙: **무결성은 생성이 아니라 게이트로 수렴한다.** 라벨(G1~G5)·엣지(G0)·순환(G6)은 **다른 축**이며 각각 막는다.
> 시드: 현재 검증된 18문항(conflict 12 / distractor 6, hop1=4·hop2=12·hop3=2)을 코어로 유지하고 확장.

---

## 0. 세 가지 무결성 축 (먼저 이해할 것)

| 축 | 무엇 | 막는 방법 | 누가 깨면 |
|----|------|-----------|-----------|
| **라벨 무결성** | gold answer/chunk/path가 맞는가 | G1~G5 | LLM 환각, 손라벨 실수 |
| **엣지 무결성** | KG 엣지 자체가 맞는가 | **G0** | REFERENCES 정규식 noise |
| **순환 무결성** | 그래프가 유리하게 "설계"된 건 아닌가 | **G6** | KG에서 뽑으면 by-construction |

> 핵심: "데이터셋 무결성 ⊆ KG 무결성." KG에 틀린 REFERENCES가 있으면 그 위 멀티홉 문항이 오류를 **상속**한다.
> CONFLICTS_WITH(전문가 큐레이션)는 깨끗하지만 REFERENCES(정규식)는 아니다 → G0로 먼저 측정.

---

## 1. 목표 분포 (층화 설계 매트릭스)

**총 80문항 목표** (60 최소 / 100 상한). 각 문항은 아래 3축 라벨을 단다.

### 축 A — hop (추론 단계)
| hop | 현재 | 목표 | 비고 |
|-----|------|------|------|
| 1 | 4 | 16 | 단일 조항 직접 |
| 2 | 12 | 40 | 두 조항 연결 |
| 3 | 2 | **24** | **★ 최우선 보강** — 멀티홉 주장의 통계력. 최소 10, 목표 24 |

### 축 B — 유형
| 유형 | 목표 | 정의 |
|------|------|------|
| 문서간 충돌 | 24 | 서로 다른 두 문서가 다른 기준 (양쪽 서술 정의 성립) |
| 단일문서 정밀 | 16 | 한 문서 내 정밀 질의 (충돌 분모서 제외) |
| 동명-무충돌 distractor | 12 | 이름 같으나 **결과 같아** 충돌 아님 (13번류) ★음성표본 |
| 용도상이 distractor | 8 | 같은 "단시간"이라도 고보 §3①(자격) vs §12⑤(규모) — 목적 다름 |
| 무관/무응답 distractor | 8 | 어떤 문서로도 답 불가 → "모름"이 정답 |
| 일반 distractor | 12 | 단순 음성 |

### 축 C — 연결 기제 (★ by-construction 분리의 핵심)
| 기제 | 목표 | 의미 |
|------|------|------|
| Concept/CONFLICTS_WITH | 30 | 전문가 큐레이션 7쌍 위 (신뢰 높음) |
| REFERENCES | 25 | 정규식 엣지 위 (G0 신뢰도 상속) |
| **held-out (앵커 밖)** | **25** | **★ KG 큐레이션 밖** — 사람이 원문서 직접 작성. 일반화 측정 |

> **C축이 가장 중요.** "생성형(앵커 위) vs held-out(앵커 밖)"을 분리해야 §6 by-construction 방어가 된다.
> held-out 25개는 특히 **hop3 + §29의8(7개 충돌앵커 밖) 연쇄류**를 다수 포함시켜 "암기 vs 일반화"를 가른다.

## 2. 생성 파이프라인 (출처별 다른 절차)

### 2.1 트랙1 — KG 반자동 생성 (생성형, ~55문항)

KG 요소에서 템플릿으로 뽑으면 **gold가 곧 샘플링한 KG 요소 그 자체**라 라벨 환각 ≈ 0(provenance-exact).

```
충돌형 질문   ← CONFLICTS_WITH 쌍 (src, dst, axis) → "A기준 vs B기준 차이?" 템플릿
멀티홉 질문   ← REFERENCES 사슬 (A→B→C) → "A에 따르면 B, B가 정한 C는?" 템플릿
단순 질문     ← 단일 조 → "X조의 기준은?" 템플릿
동명 distractor ← 같은 Concept, 다른 청크인데 결과 동일 → "양쪽 처리?" 템플릿
용도상이      ← 같은 용어 다른 조(§3① vs §12⑤) → "이 단시간은 어느 기준?"
```

**LLM은 "자연스러운 질문 문장"만 쓰게 하고, gold는 KG에서 가져온다.** LLM이 gold를 만들면 안 됨.

LLM 호출 시 **근거 제한 필수**:
```
system: "아래 [청크]에 있는 내용만으로 질문을 만들어라. 청크에 없는 조항·수치·요건을
         지어내지 마라. 질문만 출력하고 정답은 출력하지 마라."
user:   [샘플링한 청크 text] + [기대 gold = 이 청크들]
```

### 2.2 트랙2 — held-out 인적 작성 (~25문항)

KG 템플릿을 **쓰지 않고**, 사람이 원문서를 직접 읽고 작성. 목적: 순환 차단 + 일반화 측정.
- 우선순위: **hop3 + 앵커 밖 조문**(§29의8 사후관리 연쇄, 7개 충돌쌍에 없는 관계).
- 무응답 문항(정답=문서에 없음)도 여기서 작성.
- 작성자 ≠ 검수자 (G4 교차검수).

## 3. 무결성 게이트 G0~G6 (통과 못 하면 폐기)

> "생성만으로 무결"은 불가. **"게이트 통과로 무결에 수렴"**이 정확한 표현.

### G0 — 엣지 신뢰도 측정 (데이터 만들기 *전*)
- REFERENCES 엣지 랜덤 50개 샘플 → 사람이 "이 인용 진짜 맞나" 검수 → **정확도 % 산출**.
- CONFLICTS_WITH(전문가 큐레이션)와 **출처별 분리** 집계.
- 산출물: "REFERENCES 정확도 X%" → REFERENCES 기반 멀티홉 문항의 신뢰 상한으로 보고서에 명시.
- **G0 미측정 시 트랙1 멀티홉 생성 금지** (오류 상속 방지).

### G1 — Provenance (라벨 실존)
- 모든 gold_chunk_id / gold_edge가 KG에 **실존**하는지 자동 대조. 하나라도 없으면 폐기.

### G2 — 수치 Grounding (verbatim, ★단위 포함)
- gold_answer의 모든 수치를 **수치+단위 묶음**으로 청크 원문에 verbatim 존재 확인(결정론적 문자열 대조).
- ⚠️ "60시간" 단독 매칭 금지 → "월 60시간"/"주 15시간"처럼 **단위까지** 일치해야 통과.
  (우리가 계속 부딪힌 단위 오류 — 고보 월60/주15 vs 세법 월60 — 를 여기서 차단)

### G3 — 응답가능성 / 유일성
- gold_chunk만으로 정답이 **도출 가능**하고 **유일**한가 (LLM judge + 사람 스팟체크).
- 무응답 문항은 "어떤 청크로도 도출 불가 → 정답=모름"이 성립하는지 역으로 검증.

### G4 — Held-out 원문 대조 + 2인 교차검수
- 트랙2(held-out) 전량 + 트랙1 샘플 20% → 작성자 외 **2인**이 원문 대조. 불일치 시 폐기/수정.
- verified=false 의심값(군필 39세 등 세트3 계열)은 **원문 확인 전 채택 금지**.

### G5 — 누출 / 중복 차단
- 시드 18문항과 의미 중복 제거(임베딩 유사도 >0.9 폐기).
- train/eval 누출 없음(우리는 학습 안 하지만, few-shot 예시와 평가문항 겹침 금지).

### G6 — 순환 차단 (라벨과 별개 축)
- 모든 문항에 `origin ∈ {generated, held_out}` 라벨 강제.
- **보고 시 항상 두 군 분리 집계.** "전체 Recall"만 보고 금지 → "generated Recall / held-out Recall".
- held-out에서도 그래프가 이기면 = by-construction 아닌 진짜 일반화. (이게 핵심 방어)

## 4. 문항 스키마 (gold_set 확장판)

기존 필드 + 확장 라벨. jsonl 한 줄:

```json
{
  "id": 19,
  "question": "...",
  "gold_answer": "...",
  "gold_docs": ["조세특례제한법시행령"],
  "gold_articles": ["조특법시행령 제26조의8제2항제1호"],
  "gold_chunk_ids": ["조특령_제26조의8_제2항"],     // ★G1용 — 청크 실존 대조
  "gold_path": [                                      // ★멀티홉 경로(provenance gold)
    {"src":"조특령_제26조의8", "edge":"CONFLICTS_WITH", "dst":"근기령_제7조의2", "axis":"상시근로자_산정"}
  ],
  "hop": 2,
  "type": "conflict",
  "connect_mech": "CONFLICTS_WITH",                   // ★축C: Concept|REFERENCES|held_out
  "origin": "generated",                              // ★G6: generated|held_out
  "numeric_claims": [{"value":"60","unit":"시간","period":"월"}],  // ★G2용
  "answerable": true,                                 // ★G3: 무응답이면 false
  "gate_passed": ["G1","G2","G3","G5"],               // 통과 게이트 기록
  "verified_by": ["작성자", "검수1", "검수2"]          // ★G4
}
```

> 기존 18문항도 이 스키마로 마이그레이션(없는 필드는 채워넣기). `gold_chunk_ids`·`origin`이 특히 중요.

## 5. 실행 순서 (의존성)

```
1. G0 먼저 — REFERENCES 50개 검수 → 정확도 % 확정         (이거 없이 멀티홉 생성 X)
2. 시드 18개 스키마 마이그레이션 (origin=held_out로 표기)
3. 트랙1 생성 (KG 템플릿 → LLM 질문화 → 근거제한)         ~55개
4. 트랙2 작성 (사람 held-out, hop3·앵커밖 우선)            ~25개
5. 게이트 일괄 적용:
   G1(자동) → G2(자동, 단위포함) → G3(judge+스팟) → G5(중복) → G4(2인, held-out 전량)
6. 통과분만 gold_set_v2.jsonl로 동결. 폐기 로그 별도 보관.
7. 분리 집계 리포트: generated vs held_out, 기제별(Concept/REF/held), hop별.
```

## 6. 보고 시 분리 집계 (필수 — 안 하면 순환 비판 못 막음)

| 지표 | 전체 | generated | **held_out** | 해석 |
|------|------|-----------|-------------|------|
| Context Recall (G vs N) | _ | _ | **_** | held_out에서 G우위 = 진짜 일반화 |
| ┗ hop3 | _ | _ | **_** | 앵커밖 멀티홉 — H2 재검토 |
| 기제별 (Concept/REF/held) | | | | 그래프가 어느 홉에서 강한가 |
| 충돌 P/R/F1 | | | | 동명·용도상이 표본 위 |
| 무응답 회피율 | | | | Recall↔회피 교환 |

> **헤드라인은 held_out Recall로.** "앵커 위 generated"에서 이긴 건 by-construction 의심을 받지만,
> "앵커 밖 held_out"에서 이기면 그래프의 진짜 구조적 가치다. 이게 §0 순환 무결성의 최종 증명.

## 7. 현실 제약 (PM 메모)

- **게이트↔규모는 트레이드오프.** G4(2인 교차검수)를 강화할수록 처리 속도가 준다.
- 80개 목표지만, **G통과 못 하면 60개라도 무결한 게 100개 중 라벨 깨진 것보다 낫다.**
- 우선순위가 흔들리면: **hop3 held-out > 동명/용도상이 distractor > 나머지.**
  (hop3 표본과 held-out이 현재 가장 비어있고, 발표 주장에 가장 크리티컬)
- 시드 18개는 **절대 버리지 말 것** — 이미 원문 검증된 코어.

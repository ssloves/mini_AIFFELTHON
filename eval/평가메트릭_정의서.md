# 평가 메트릭 정의서 (Legal GraphRAG)

> qa.md/gold_set.jsonl이 **"정답지"**라면, 이 문서는 **"채점 방법 + 실행 가이드라인"**이다.
> 가이드 §6.2의 지표를 **우리 실제 아키텍처(Step 4 에이전트)에 맞춰** 계산 가능한 공식으로 정의한다.
> 구현(현행): `agent/run_eval.py`. 실행 절차는 §9 평가 가이드라인 참조.

> **이 문서가 정의서 v1에서 바뀐 점(아키텍처 정렬):**
> 1) 두 비교 시스템을 우리 구현으로 확정 — GraphRAG=Step4 에이전트, Naive=**강화 벡터 RAG**(예산 일치+조문 다양화).
> 2) Context Recall 매칭 단위를 **조(條) 단위로 확정**(우리 청크 최소 단위가 조 단위 → 항/호 정밀채점은 구조상 상한).
> 3) Conflict Detection 분모를 **cross-doc 충돌 7개로 재정의**(단일법령 문항 5개는 측정 대상에서 분리).
> 4) 정직성 라벨 규칙(hop3 n=2, 충돌 recall의 구성상 우위) 명문화.

---

## 0. 핵심 구분 — 이 프로젝트가 *증명하려는 것*

> **주장: "GraphRAG는 더 똑똑한(정확한) 답이 아니라, 더 신뢰할 수 있는(근거를 안 놓치는) 답을 낸다."**

따라서 **1순위 지표는 Answer Accuracy가 아니라 Context Recall**이다.
"답을 맞혔는가"보다 "정답 근거 조항을 실제로 검색해 왔는가"가 우리 가설의 핵심.

| 우선순위 | 지표 | 무엇을 증명 |
|---------|------|------------|
| ★★★ | **Context Recall** | 그래프 확장이 근거 수집률을 높인다 (핵심 가설) |
| ★★★ | **Hop별 누락률** | hop이 깊을수록 Naive가 무너지고 Graph는 유지된다 (대표 차트) |
| ★★ | **Conflict Detection** | CONFLICTS_WITH 엣지의 가치 (분모=cross-doc 충돌 7개) |
| ★★ | **Conflict 오탐률** | distractor에서 충돌을 잘못 외치지 않는가 (과잉탐지 점검) |
| ★ | **Faithfulness** | 환각 없는가 (Verifier 효과) |
| ★ | **Answer Accuracy** | 최종 정답률 (보조 지표) |

### 0.1 비교 대상 두 시스템 (우리 구현 확정)

| | **GraphRAG (Ours)** | **Naive RAG (강화 베이스라인)** |
|---|---|---|
| 검색 | Router→Retriever: 벡터 시드 + **개념앵커**(CONFLICTS_WITH 정준쌍) + 개념브리지 + 참조 확장 → 재랭킹 | **순수 벡터**: over-fetch 후 같은 조문 중복 제거(다양화) |
| 청크/임베딩 | 동일 청크·동일 임베딩(text-embedding-3-small) | **동일** — 검색 방식만 다르게(변인 통제) |
| 예산(청크 수) | 라우트별 최종 컨텍스트(6~12, 재시도 +4) | **그래프와 동일 예산**(최소 8 보장) |
| 추론 | Reasoner→Verifier(근거·수치 검증, 회귀 루프) | 단일 LLM 호출(그래프·앵커·Verifier 미사용) |
| 구현 | `agent/app.py` | `agent/run_eval.py: baseline_context()` + naive 생성 함수 |

## 1. Context Recall (★ 핵심)

### 정의
검색기(retriever)가 **gold 근거 조항에 해당하는 청크를 실제로 수집했는가**의 비율.

```
                수집한 청크 중 gold 조항에 매칭되는 것의 수
Context Recall = ─────────────────────────────────────────
                       해당 질문의 gold 조항 총 수
```

### 입력
- `gold_articles` (gold_set.jsonl): 이 질문의 정답 근거 조항 리스트
- 각 시스템이 반환한 컨텍스트 청크들의 `(doc, article)` — GraphRAG=최종 컨텍스트, Naive=강화 베이스라인 결과

### 매칭 단위 — **조(條) 단위로 확정**
우리 청크의 최소 단위가 **조(條)**이므로(예: `조특법시행령 제26조의8` 한 덩이), 채점도 조 단위로 한다.
- gold `"조특법시행령 제26조의8제2항제2호"` → 정규화 → `(조특법시행령, 제26조의8)`
- 청크 메타 `doc=조특법시행령, article=제26조의8` 과 비교 → **조가 일치하면 hit**(항/호는 안 따짐).
- 정규화 함수: `제N조[의M]`만 추출(공백·항·호·괄호 제거). 별도 `article_map.json` 없이 **정규식 1개로 해결**.
- `장려금지침`은 조 구조가 없으므로 gold 토큰을 `장려금지침|*`(문서 단위)로 두고 지침 청크면 hit.

### 계산 (문항 단위 → 평균)
```
질문 q의 gold = {조특법시행령 제26조의8, 고용보험법시행령 제3조}   (조 단위 2개)
시스템 수집 청크의 조 = {고용보험법시행령 제3조, 근로기준법시행령 제7조의2}
매칭 = {고용보험법시행령 제3조}                                  (1개)
Context Recall(q) = 1/2 = 0.5
```
전체 점수 = 모든 문항의 평균(시스템별).

### ⚠️ 정직성 메모 — 조 단위 매칭의 한계와 충돌 recall의 "구성상 우위"
- **조 단위라 느슨하다**: `제26조의8` 안의 어느 항이든 들어오면 hit. 즉 세법 상시근로자(`§26의8`)는
  거의 모든 충돌의 한쪽이라 한 번 들어오면 여러 문항이 동시에 채워진다 → **recall이 다소 높게 보일 수 있음**.
- **cross-doc 충돌 문항의 GraphRAG recall은 부분적으로 "구성상(by-construction)"**: 우리 개념앵커가
  주입하는 조문(CONFLICTS_WITH 양끝)이 곧 그 문항의 gold이기도 하다. 발표 시 "이 우위는 그래프가
  *설계상* 정준 충돌쌍을 끌어오기 때문"이라고 정직하게 프레이밍할 것(약점이 아니라 설계 의도).

### 해석
- Graph > Naive 이면 가설 지지.
- distractor 문항은 gold가 1개라 둘 다 높게 나옴 → 변별력은 **hop2~3 cross-doc 충돌 문항**에서 나온다.

## 2. Hop별 누락률 (★ 대표 차트)

### 정의
Context Recall을 **hop 그룹별로 분리**해 본 것. 누락률 = 1 − Context Recall.

```
누락률(hop=h) = 1 − (hop이 h인 문항들의 평균 Context Recall)
```

### 계산
gold_set.jsonl의 `hop` 필드로 문항을 1/2/3 그룹으로 나누고, 각 그룹에서 Naive·Graph의 평균 누락률.

### 출력 (대표 차트 — 꺾은선/막대)
```
누락률(%, 낮을수록 좋음)
            hop1     hop2     hop3
Naive RAG    a        b        c     ← hop 깊을수록 급증해야 가설 성립
GraphRAG     d        e        f     ← 완만하게 유지
            (값은 측정으로 채움)
```

### ⚠️ 표본 경고
현재 hop1=4, hop2=12, **hop3=2**. hop3이 2개면 차트의 가장 중요한 점이 표본 2개.
→ **발표 전 hop3를 4~5개로 보강 권장** (검증된 연쇄추징·출산휴가→추징 시나리오 변형).
→ 보강 전이면 차트에 "hop3 n=2 (참고)" 명시 — 정직성.

### 해석
두 선의 **간격이 hop 따라 벌어지면** = 우리 핵심 주장 증명. hop1에서 비슷한 건 정상(오히려 정직).

## 3. Conflict Detection (★ 충돌)

### 정의 (분모 재정의 — **cross-doc 충돌 7개**)
`type=="conflict"` 12개 중 **5개(Q4·Q5·Q6·Q9·Q12)는 단일 법령 문항**이라 "양쪽 기준 서술"이
성립하지 않는다(서로 다른 두 문서가 없음). 따라서 Conflict Detection은 **cross-doc 충돌 7개
(Q1·Q2·Q3·Q7·Q8·Q10·Q11)** 만 분모로 한다.

```
                  두 문서 기준을 모두 서술+차이 명시한 cross-doc 충돌문항 수
Conflict Detection = ──────────────────────────────────────────────────
                            cross-doc 충돌 문항 수 (=7)
```

> 단일법령 5개는 "**단일법령 정밀질의**"로 분류해 **Context Recall·Answer Accuracy로만 평가**하고
> Conflict Detection에서는 제외한다(0점 처리 금지 — 측정 대상이 아님).

### 우리 시스템의 산출 신호
- GraphRAG는 retriever가 CONFLICTS_WITH 엣지를 컨텍스트에 넣으면 `conflict_flagged=true` +
  `conflict_desc`를 낸다. 이 둘이 채점 입력.
- 7개 cross-doc 충돌은 우리 **CONFLICTS_WITH 엣지 7개와 1:1 대응**하므로 커버리지 매핑이 자동으로 맞는다.

### 채점 방법 (2단계)
1. **구조 1차**: 답변에 양쪽 문서 근거 조가 다 인용됐고 `conflict_flagged=true`인가.
2. **LLM-judge 2차**: "이 답변이 두 제도의 기준 차이를 명시했는가? (yes/no)" — gold_answer 기준.

### ⚠️ Conflict 오탐률 (distractor 과잉탐지 — 우리 구조의 핵심 시험대)
`type=="distractor"` 6개에서 충돌을 잘못 외치면 감점:
```
충돌 오탐률 = distractor인데 conflict_flagged=true인 수 / 전체 distractor 수 (=6)
```
- **특히 Q13(월50h: "표현 다르나 결과 같음")이 우리 아키텍처의 진짜 시험**이다. 질문에 `단시간`이
  잡히고 비교형이라 라우터가 conflict로 분류되면 개념앵커가 충돌쌍을 주입 → `conflict_flagged=true`로
  **과잉탐지**할 수 있다. 이 케이스가 틀리면 그것이 곧 "규칙 기반 충돌 게이팅은 *명목충돌*과
  *실질충돌(결과 동일)*을 구분 못 한다"는 **정직한 한계 발견**이다(숨기지 말고 보고).
- 충돌은 잡고(7/7 지향) distractor엔 안 속는(오탐률↓) 2×2가 진짜 신뢰성 슬라이드.

## 4. Faithfulness (환각 없음)

### 정의
답변의 모든 주장이 **수집된 컨텍스트 안의 근거로 뒷받침**되는가 (= parametric memory로 지어내지 않았는가).

```
                답변 주장 중 컨텍스트로 뒷받침되는 것의 수
Faithfulness = ──────────────────────────────────────
                        답변의 전체 주장 수
```

### 채점 방법
- LLM-judge: 답변을 문장(주장) 단위로 쪼개, 각 주장이 컨텍스트에서 근거를 찾을 수 있는지 yes/no.
- 우리 Verifier는 ① **근거 존재성**(인용 chunk_id가 컨텍스트에 있는가, 조 단위 정규화로 보정)과
  ② **수치 충실성**(원문과 모순 없는가)을 본다(충돌 커버리지·과잉탐지는 결정론 가드가 별도 처리).
- **Verifier ON/OFF 비교**가 곧 이 노드의 효과 측정: OFF면 Reasoner 1차 답을 그대로, ON이면 회귀
  루프를 거친 최종 답을 채점. (구현상 OFF는 verifier 노드를 bypass하는 플래그로.)

### 해석
이 지표는 ★ 보조. 시간이 부족하면 ON/OFF 비교는 부록으로 미룬다.

### 4.1 근거 귀속(grounding) 사실성 3지표 — 로드맵 2번 산출(결정론 강화)

기존 LLM 단일 Faithfulness judge는 잡음이 크고 재현성이 낮다. 이를 **근거 귀속 기반 3지표**로 분해한다
(구현: `eval/score_factuality.py`, 저장 답변 재채점 — 모델 무수정).

| 지표 | 방식 | 정의 |
|------|------|------|
| (A) **수치 근거율** | 결정론 | 답변의 *의미있는 수치*(세/시간/년/개월/명/원/%)가 인용 청크 text에 실존하는 비율. 환각=근거 없는 수치. |
| (B) **엄격 LLM 지지도** | LLM(보조) | **인용 청크 text에만** 근거해 핵심 주장 지지 정도(모수지식 사용 금지). (A)가 못 잡는 정성 주장 보완. |
| (C) **인용 해소율** | 결정론 | 답변의 `[chunk_id]`가 KG에 존재(조 단위 폴백 후)하는 비율. 폴백으로도 못 찾으면 *미해소(환각 의심)*. |

**채점 전제(이게 없으면 *위양성*이 난다 — 로드맵 2에서 실증).**
- **(전제 1) 인용 조 단위 정규화**: 답변이 `[고보령_제3조_제1항]`으로 적어도 실제 청크가 `…_제1-2항`이면
  같은 *조(條)* 로 보고 근거를 인정한다(Verifier의 `_article_key`와 동일 규칙). 정확 일치만 보면 정상 답을
  환각으로 오탐한다.
- **(전제 2) 질문 수치 화이트리스트**: 답변이 **질문문에 등장한 전제 수치**(예: "상시 4.8명", "55시간")를
  복창한 것은 환각이 아니다 → 근거 텍스트에 *질문문을 포함*해 채점한다.
- **(전제 3) 연도 토큰 제외**: `2024/2025년` 같은 연도는 수량 수치가 아니므로 (A)에서 제외한다.

**해석 원칙**: 환각의 *결정론적* 증거는 (A)·(C)이며, (B)는 보조다. "근거 없는 정답"을 골라내는 본 지표는
Accuracy(맞았나)와 직교한다("맞았나"와 "근거 있게 맞았나"의 분리).

## 5. Answer Accuracy (보조)

### 정의
최종 답변이 `gold_answer`와 **의미적으로 일치**하는가 (LLM-as-a-Judge, 0~1 또는 yes/no).

### 채점 방법
```
judge_prompt = "다음 [모범답안]과 [시스템답변]이 핵심 결론·근거에서 일치하는가? 0~1 점수로."
점수 = LLM(judge_prompt, gold_answer, system_answer)
```

### ⚠️ 왜 보조인가
LLM이 **근거 없이도 우연히 답을 맞힐 수 있다**(예: 사전학습 지식으로 "차감됩니다"라고 답). 그래서
정답률만 보면 GraphRAG 우위가 안 드러날 수 있다. → 우리 주장은 Context Recall이 1순위, Accuracy는 참고.

## 6. 비교 실험 매트릭스 (최종 산출물)

| 지표 | Naive RAG(강화) | GraphRAG | 우위 해석 |
|------|-----------|----------|-----------|
| Context Recall (전체, 조 단위) | _ | _ | 높을수록 ★ |
| ┗ hop1 / hop2 / hop3 | _/_/_ | _/_/_ | **간격 벌어짐이 핵심** (hop3 n=2 참고) |
| Conflict Detection (분모=7) | _ | _ | cross-doc 충돌 감지율 |
| Conflict 오탐률(distractor, 분모=6) | _ | _ | 낮을수록 ★ (Q13 핵심) |
| Faithfulness | _ | _ | Verifier ON/OFF 효과 |
| Answer Accuracy | _ | _ | 보조 |

> Naive는 강화판(예산 일치 + 조문 다양화)이므로 "약한 대조군" 반박을 피한다.
> Conflict/Faithfulness/Accuracy의 Naive 열을 채우려면 Naive **생성 함수**(단일 LLM 호출)가 필요.

> 발표 핵심 슬라이드 2장:
> ① **Hop별 누락률 꺾은선** (Naive 급증 vs Graph 유지) — 주장 증명
> ② **Conflict 감지 vs 오탐 2×2** (충돌은 잡고 distractor엔 안 속음) — 신뢰성 증명

---

## 7. 실행 흐름 (의사코드 — 우리 구현 기준)

```
norm(article) = "제N조[의M]"만 추출(공백·항·호·괄호 제거)   # article_map 불필요

for q in load("gold_set.jsonl"):           # 정본 18문항
    # GraphRAG = Step4 에이전트(검색+추론+검증)
    g = run(q.question)                     # agent/app.py  → context, answer, conflict_flagged
    # Naive = 강화 벡터(예산=g.context 수) + 단일 LLM 생성
    n_ctx = baseline_context(q.question, budget=len(g.context))
    n_ans = naive_generate(q.question, n_ctx)        # 그래프/앵커/Verifier 미사용

    for sys, ctx, ans, flagged in [("graph", g.context, g.answer, g.conflict_flagged),
                                   ("naive", n_ctx,    n_ans,    said_conflict(n_ans))]:
        # 1) Context Recall (조 단위)
        hit = sum(covers(ctx, t) for t in q.gold_articles)
        recall[sys][q.hop].append(hit / len(q.gold_articles))

        # 2) 충돌 (cross-doc 충돌 7개만 분모 / distractor는 오탐률)
        if q.id in CROSS_DOC_CONFLICT:   # {1,2,3,7,8,10,11}
            conflict[sys].append(judge_conflict(ans, q.gold_answer) and flagged)
        elif q.type == "distractor":
            false_alarm[sys].append(flagged)
        # 단일법령 충돌 5개(4,5,6,9,12)는 충돌 채점 제외 — Recall/Accuracy로만

        # 3) Faithfulness / Accuracy (선택)
        faith[sys].append(judge_faith(ans, ctx))
        acc[sys].append(judge_acc(ans, q.gold_answer))

report = aggregate(recall, conflict, false_alarm, faith, acc)   # hop별 평균 등
```

> 현행 `agent/run_eval.py`는 1·2(오탐 포함)·hop_ok·리포트까지 구현돼 있다. 정본 실행을 위해
> 추가할 것: **18문항 정본 gold_set + hop별 Recall 집계 + Naive 생성 함수 + (선택)Faithfulness/Accuracy judge.**

## 8. 측정 우선순위 (시간 배분)

| 순위 | 측정 | 필요 조건 | 못 하면 |
|------|------|-----------|---------|
| 1 | Context Recall + hop별 | 두 시스템 검색 + norm() | 발표 불가 (필수) |
| 2 | Conflict 감지/오탐 | CONFLICTS_WITH 엣지(있음) + cross-doc 7 매핑 | 충돌 슬라이드 빠짐 |
| 3 | Faithfulness | Verifier ON/OFF | 부록으로 |
| 4 | Answer Accuracy | Naive 생성 + LLM judge | 부록으로 |

> **최소 발표 성립선: 1번(Context Recall hop별)만 나와도 핵심 주장은 증명된다.**
> 2번까지 나오면 충돌 슬라이드가 살고, 3·4는 시간 남으면.

- 리포트: `agent/eval_report.md`(자동). 발표용 수치는 여기서 가져온다.

### 9.3 산출물 → 슬라이드 매핑
| 산출물 | 출처 | 슬라이드 |
|--------|------|---------|
| Hop별 누락률(1−Recall) 꺾은선 | hop별 평균 Recall(G vs N) | ① 핵심 주장 |
| 충돌 감지 vs 오탐 2×2 | Conflict Detection(7) + 오탐률(6) | ② 신뢰성 |
| 문항별 표 + 한 건 trace | `eval_report.md` + `app.py` trace | 부록/시연 |

### 9.4 정직성 라벨 규칙 (발표·리포트에 반드시 표기)
1. **hop3 n=2 (참고)**: 핵심 차트의 hop3 점은 표본 2개. 보강 전이면 차트에 명시.
2. **충돌 recall의 구성상 우위**: cross-doc 충돌의 GraphRAG Recall 우위는 개념앵커가 정준 충돌쌍을
   *설계상* 끌어오기 때문 — "성능"이 아니라 "설계 의도"로 정직하게 설명.
3. **조 단위 채점**: 항/호 정밀도는 청크 구조상 상한(조 단위)이라는 점 명시.
4. **Q13 결과는 숨기지 않는다**: 과잉탐지가 나오면 그 자체가 "명목충돌 vs 실질충돌 미구분"이라는 한계 발견으로 보고.
5. **사실성 채점 위양성 방지(로드맵 2)**: §4.1의 (전제 1) 인용 조 단위 정규화 · (전제 2) 질문 수치
   화이트리스트 · (전제 3) 연도 제외를 적용한 *뒤* 환각을 집계한다. 미적용 시 정상 답을 환각으로 오탐한다.
6. **'오염' 주장은 청크 원문 대조 후에만**: 특정 수치를 "미확인/오염"으로 단정하기 전, 반드시 인용 청크
   원문을 대조한다(17번 "군필 39세"는 `지침_p29_s64`에 실재 → 오염 아님으로 정정된 전례).

### 9.5 변인 통제 규칙 (반박 방어)
- 두 시스템은 **같은 청크·같은 임베딩·같은 예산**. 다른 것은 오직 "검색 방식(그래프 확장 유무)"과
  "추론 파이프라인(Verifier 유무)"뿐. → 차이가 "그래프 효과"임을 입증.
- Naive는 강화판(over-fetch + 조문 다양화)이라 "약한 대조군" 비판을 차단.

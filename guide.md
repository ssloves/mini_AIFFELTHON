## 2. Step 1 — 데이터 정제 (PDF → Markdown)

### 2.1 왜 마크다운인가

PDF는 **"보이는 대로"** 저장하는 포맷이다. 텍스트의 논리적 구조(제목, 조항, 항, 호의 위계)가 아니라 **좌표와 폰트 크기**로 레이아웃을 표현한다. 이 때문에:

- 같은 "제29조의8"이라는 조항이 PDF에서는 단순히 "특정 좌표의 텍스트 블록"일 뿐이고, 그 아래 제1항~제8항이 어디서 시작하고 끝나는지가 기계적으로 명확하지 않다.
- 표(table)는 좌표 기반 텍스트 배치로 표현되어, 행/열 구조가 추출 과정에서 깨지기 쉽다.
- 머리글/바닥글(페이지 번호, "법제처 국가법령정보센터" 등)이 본문과 섞인다.

**마크다운으로 변환하는 이유는 다음 단계(청킹·그래프 구축)에서 문서의 논리적 위계를 활용하기 위해서다.** 마크다운의 `#`, `##`, `###` 헤딩이 법률 문서의 조-항-호 구조와 자연스럽게 대응되며, 이 위계 정보가 곧 그래프의 CONTAINS 엣지가 된다.

### 2.2 마크다운 구조 설계 원칙

법률 문서의 구조적 특성에 따라 마크다운을 **다음과 같은 위계**로 정리한다:

```
# [문서 제목] ← 법률명/지침명 (예: 조세특례제한법)
## [편/장 제목] ← 있는 경우만 (예: 제3장 고용지원)
### [조 제목] ← 핵심 단위 (예: 제29조의8 통합고용세액공제)
#### 제N항 ← 조 아래의 각 항
- 제N호 ← 항 아래의 각 호
  - 가. 목 ← 호 아래의 각 목
```

**이렇게 설계하는 이유:**

1. **조(Article)가 그래프의 핵심 노드 단위**가 된다. 법률 해석에서 "제29조의8에 따르면..."이라고 인용하는 것처럼, 조가 법률 추론의 기본 참조 단위이다.
2. **항(Paragraph)이 청킹의 자연스러운 경계**가 된다. 하나의 조 안에서도 항마다 규율하는 내용이 다르기 때문에 (예: 제6항은 사후관리, 제8항은 위임 조항) 항 단위로 청킹해야 검색 정밀도가 올라간다.
3. **호(Subparagraph)와 목(Item)은 항에 종속**되므로, 항 청크 안에 포함시킨다. 호만 단독으로 떼어내면 맥락이 사라진다.

### 2.3 문서 유형별 정제 주의사항

| 문서 유형 | 핵심 주의점 |
|----------|-----------|
| **법률/시행령** (조특법, 고용보험법시행령, 근로기준법시행령) | 조-항-호-목 위계를 정확히 파싱. 부칙, 별표는 별도 섹션으로 분리. 개정일자 (`<개정 2024. 12. 31.>`) 메타데이터로 보존 |
| **행정지침** (장려금지침) | 장-절-항 대신 "유형Ⅰ/Ⅱ", "장", "절" 구조. 별첨(별첨1~5)은 본문과 별도로 처리하되, 본문에서 "☞ [별첨1]"로 참조하는 관계를 메타데이터로 기록 |

### 2.4 정제 시 보존해야 할 메타데이터

마크다운 변환 시 단순 텍스트만 남기면 안 된다. **다음 메타데이터를 YAML frontmatter 또는 인라인 태그로 보존**해야 그래프 구축에서 활용할 수 있다:

- **문서 식별자**: 법률명, 법률 번호, 시행일
- **조항 식별자**: 조 번호, 항 번호 (예: `article: 29조의8`, `paragraph: 6`)
- **개정 이력**: 최종 개정일 (시간적 유효성 판단에 필요)
- **상호 참조**: 본문 내에서 다른 조항을 인용하는 부분 (예: "제127조에 따라", "동법 시행령 제26조의8") — 이것이 그래프의 REFERENCES 엣지의 원천이 된다

### 2.5 이 단계의 최종 결과물

```
data/processed/
├── 장려금지침.md           # 191p, 행정지침(유형/장/절 + 표 355개 + 페이지앵커 188개)
├── 고용보험법시행령.md      # 63p,  조 222 · 항 419 · 호 603 · 목 180
├── 근로기준법시행령.md      # 15p,  조 81 · 항 74 · 호 203 (장 구분 없는 평면 구조)
├── 조특법.md               # 255p, 장 20 · 조 538 · 항 1797 · 호 1541 · 목 563
└── 조특법시행령.md          # 351p, 장 20 · 조 508 · 항 2449 · 호 2582 · 목 756
```

> 실제 산출 통계(2026-06 기준). 모든 파일에 YAML frontmatter(법령명/번호/시행일/개정구분),
> `<!-- page: N -->` 페이지 앵커, `<개정 …>`·`삭제<…>` 태그, 부칙 섹션이 보존되어 있다.

### 2.6 정제 파이프라인 구현 및 추가 전략 (실측 기반)

Step 1은 `notebooks/01_pdf_to_md.py` 단일 스크립트로 구현했다(`python notebooks/01_pdf_to_md.py`).
원본 PDF 5종을 직접 점검한 결과를 바탕으로, **가이드 §2.1~§2.4의 요구사항에 더해 다음 전략을
추가**했다. 모두 "데이터를 실제로 본 뒤" 결정한 것이다.

**(0) 추출 엔진 선택 — pdfplumber 채택**

`pdfplumber` vs `pypdfium2`를 동일 페이지로 비교한 결과, pdfplumber가 법령명 묶음표(`「」`)와
요건 마커(`➍ ❶ ❿ √ ○`)를 더 충실히 보존했다. 4개 법령은 텍스트가 깨끗하고 표가 없어
선형 추출로 충분했고, 장려금지침만 표가 많아(191p 중 99p) 표를 별도 처리했다.

**(1) 논리 줄 복원 (De-wrapping) — 가장 중요한 추가 전략**

PDF는 한 항/호가 물리적으로 여러 줄로 끊겨 추출된다. 이를 그대로 두면 `제127조`,
`상시 4.8명`, `<개정 …, 2023.` + `11. 7.>` 같은 **참조·수치·개정일이 줄 경계에서 분절**되어
Step 2의 참조 정규식과 청킹이 깨진다. 따라서 선두 마커(조/항/호/목/장)가 없는 줄은 직전
논리 단위에 병합한다. 특히 **닫히지 않은 `<개정 …` 태그가 줄바꿈된 경우**는 마커처럼 보여도
(예: `11. 7.>`) 강제로 연속행 처리한다.

**(2) 페이지 앵커 보존 (`<!-- page: N -->`) — 평가 연동용**

QA gold_source가 장려금지침을 **페이지 단위로 인용**한다(p.13, p.25, p.48, p.53 등). 모든 문서에
페이지 앵커를 삽입해 두면 §6의 Context Recall 채점 시 "조항/페이지 → 청크" 매핑이 가능해진다.

**(3) 머리글/바닥글 제거**

매 페이지 반복되는 문서 제목(`고용보험법 시행령`), 바닥글(`법제처 / 국가법령정보센터`),
단독 페이지번호를 제거한다. 1페이지의 시행정보(`[시행 …] [대통령령 제…호, …]`)는 파싱하여
frontmatter로 올린 뒤 본문에서 제외한다.

**(4) 삭제 조항·개정 태그 보존**

`제5조 삭제 <2021. 2. 17.>`처럼 삭제된 조항도 그대로 남긴다. 조 번호에 결번이 생겨도
"누락"이 아니라 "삭제"임을 명시하기 위함이며, `<개정/신설/본조신설 …>` 태그는 조항의 시간적
유효성 판단(어느 시점 법령인지)에 쓰이므로 인라인 보존한다.

**(5) 행정지침(장려금지침) 전용 처리**

- 표지(1p)는 CID 폰트라 텍스트 추출이 불가능 → **스킵**.
- 조-항-호 위계가 없으므로 유형(`유형Ⅰ/Ⅱ`)·장·절(`1-4`)을 헤딩으로, 요건 블록(`➀ ❶ ○ √`)을
  리스트로 매핑.
- 표는 `page.extract_tables()`로 추출해 마크다운 표(`[표 pN-i]`)로 렌더(선형 추출 시 열 순서가
  섞이는 업종코드 별첨 표 대응).
- `☞ [별첨N]` 참조는 본문에 그대로 보존 → Step 3에서 REFERENCES 엣지 후보로 활용.

> **한계 및 future work**: 장려금지침은 반정형 레이아웃이라 일부 줄바꿈/표 셀 정렬이 완벽하지
> 않다(목차 점선, 다단 표 등). 4개 법령 대비 후처리 보정 여지가 있으며, 필요 시 LLM 기반
> 표 정규화를 Step 2에서 보강할 수 있다.

## 3. Step 2 — EDA 및 그래프 스키마 설계

### 3.1 EDA의 목적

EDA(탐색적 데이터 분석)의 목적은 단순히 "데이터를 살펴보는 것"이 아니다. **그래프를 어떤 입도(granularity)로 구축할지 결정**하기 위한 분석이다. 구체적으로:

1. **청킹 단위 결정**: 조 단위? 항 단위? 문장 단위? — 각 단위의 평균 토큰 수와 의미적 자족성(self-containedness)을 분석
2. **상호 참조 패턴 파악**: 어떤 조항이 어떤 조항을 참조하는지 빈도와 패턴 분석 — 이것이 REFERENCES 엣지의 밀도를 결정
3. **핵심 엔티티 추출**: 법률 문서에서 반복적으로 등장하는 핵심 개념(청년, 상시근로자, 세액공제, 피보험자 등) 식별 — 이것이 Concept 노드가 된다
4. **문서 간 충돌 지점 식별**: 같은 용어(예: "상시근로자", "청년")가 문서마다 다르게 정의되는 지점 — 이것이 CONFLICTS_WITH 엣지의 후보가 된다

### 3.2 청킹 전략 — "법률 구조 인식 청킹"

일반적인 RAG에서는 고정 길이(예: 512토큰)로 텍스트를 자르거나, 문단 단위로 나누는 방식을 쓴다. **법률 문서에서는 이 방식이 치명적이다.** 이유:

- **고정 길이 청킹의 문제**: "제29조의8 ⑥ 제1항에 따른 세액공제를 적용받은 사업자가..."라는 문장이 512토큰 경계에서 잘리면, 뒤에 오는 "3년 이내에 ... 감소한 경우에는 ... 납부하여야 한다"라는 핵심 결론이 다른 청크로 가버린다.
- **문단 단위 청킹의 문제**: 법률은 하나의 항이 수백 단어에 달하는 단일 문장인 경우가 많다. 문단 구분이 의미 단위와 일치하지 않는다.

**권장: 법률 구조 인식 청킹(Structure-Aware Chunking)**

```
청킹 단위 결정 기준:
├── 법률/시행령 → "항(Paragraph)" 단위
│   ├── 항이 300토큰 미만이면 → 같은 조의 인접 항과 병합
│   ├── 항이 1500토큰 초과이면 → 호(Subparagraph) 단위로 분할
│   └── 각 청크에 상위 조(Article) 제목을 컨텍스트로 첨부
│
└── 행정지침(장려금지침) → "소절 + 요건 블록" 단위
    ├── 표(table)는 하나의 청크로 보존 (표를 쪼개면 의미 소실)
    ├── "요건 내용" 블록 (➀, ➁, ➂ 등)은 하나의 청크로 묶기
    └── 각 청크에 상위 장/절 제목을 컨텍스트로 첨부
```

**각 청크에 반드시 포함해야 할 필드:**

| 필드 | 설명 | 예시 |
|------|------|------|
| `chunk_id` | 고유 식별자 | `조특법_29조의8_6항` |
| `source_doc` | 원본 문서명 | `조세특례제한법` |
| `article` | 소속 조항 번호 | `제29조의8` |
| `paragraph` | 소속 항 번호 | `제6항` |
| `hierarchy_path` | 위계 경로 | `조특법 > 제3장 > 제29조의8 > 제6항` |
| `text` | 청크 본문 | (실제 법문) |
| `references` | 본문 내 타 조항 참조 목록 | `["제132조", "시행령 제26조의8"]` |
| `keywords` | 핵심 개념어 | `["세액공제", "상시근로자", "추징"]` |

### 3.3 그래프 스키마 설계 — 노드(Node) 정의

법률 GraphRAG에서 노드는 **5가지 유형**으로 설계한다:

| 노드 레이블 | 의미 | 속성 | 인스턴스 예시 |
|------------|------|------|------------|
| `Document` | 하나의 법률/지침 문서 | `name`, `law_number`, `effective_date` | "조세특례제한법" |
| `Article` | 법률의 "조" 단위 | `number`, `title`, `full_path` | "제29조의8(통합고용세액공제)" |
| `Chunk` | 실제 검색·추론 단위 (항/요건 블록) | `chunk_id`, `text`, `embedding`, `token_count` | "제29조의8 제6항" |
| `Concept` | 법률 전체에 걸쳐 등장하는 핵심 개념 | `name`, `definition_sources` | "상시근로자", "청년", "세액공제" |
| `Condition` | 요건/조건 블록 (장려금지침 특화) | `type`, `description`, `applies_to` | "❹ 5인 미만 기업 참여 조건" |

**왜 Concept 노드가 필요한가?**

"상시근로자"라는 용어는 조특법시행령 제26조의8, 고용보험법시행령, 근로기준법시행령에서 **각각 다르게 정의**된다. Concept 노드 없이 텍스트 검색만 하면 "상시근로자"를 검색했을 때 3개 문서의 청크가 뒤섞여 나오고, 어떤 정의가 어떤 맥락에서 적용되는지 구분할 수 없다. Concept 노드를 두면 "상시근로자" → 각 문서별 정의 청크로의 명시적 연결이 생기고, 정의 간 차이(충돌)도 엣지로 표현할 수 있다.

### 3.4 그래프 스키마 설계 — 엣지(Edge) 정의

| 엣지 타입 | 시작 → 끝 | 의미 | 생성 방법 |
|----------|----------|------|----------|
| `CONTAINS` | Document → Article → Chunk | 포함 관계 (위계 구조) | 마크다운 헤딩 파싱으로 자동 생성 |
| `REFERENCES` | Chunk → Chunk | 법문 내 상호 인용 ("제127조에 따라") | 정규식으로 "제N조" 패턴 추출 후 매핑 |
| `DEFINES` | Chunk → Concept | 이 청크가 해당 개념을 정의함 | 키워드 + LLM 보조 추출 |
| `MENTIONS` | Chunk → Concept | 이 청크가 해당 개념을 언급함 (정의 X) | 키워드 매칭 |
| `CONFLICTS_WITH` | Chunk ↔ Chunk | 동일 개념에 대해 다른 기준 적용 | EDA에서 식별 + 수동 라벨링 |
| `DELEGATES` | Document → Document | 법률→시행령 **위임(구체화)** 관계 *(우선적용 아님)* | 법률 위계 규칙 기반 |
| `NEXT_PARAGRAPH` | Chunk → Chunk | 같은 조 내 인접 항 순서 | 항 번호 순서로 자동 생성 |

**엣지 중 가장 중요한 것: REFERENCES와 CONFLICTS_WITH**

- **REFERENCES**: 법률은 "제127조에 따라", "동법 시행령 제26조의8에 따라" 같은 상호 인용이 극도로 빈번하다. 이 인용 관계를 엣지로 명시하면, 질문에 대해 **단일 조항만 검색하지 않고 인용 사슬을 따라 관련 조항을 연쇄 탐색**할 수 있다. 이것이 GraphRAG가 일반 RAG보다 multi-hop 추론에 강한 핵심 이유이다.
- **CONFLICTS_WITH**: QA셋에서 "충돌있음"으로 표기된 질문들이 이 엣지를 통해 탐지된다. 예를 들어 조특법시행령의 "상시근로자" 정의와 장려금지침의 "피보험자 수" 산정법이 다르다는 것을, 두 청크 사이의 CONFLICTS_WITH 엣지로 표현한다.

### 3.5 이 단계의 최종 결과물

```
data/
├── chunks/all_chunks.jsonl         # 3,902 청크 (chunk_id, text, metadata, 타입드 references, keywords, defines)
├── concepts.json                   # 개념 인벤토리 20종 + 문서별 멘션/정의 위치
├── conflict_candidates.jsonl       # CONFLICTS_WITH 후보 70쌍 + 규칙 스코어
├── conflicts_confirmed.jsonl       # 정준 조문 앵커링 확정 7건(axis/purpose/note/verified)
├── conditions.jsonl                # 지침 Condition 158건
├── eda_stats.json                  # EDA 수치(JSON)
└── graph_schema.md                 # 노드/엣지 정의 + 실제 인스턴스 수 + 생성방법
eda_report.md                       # EDA 리포트(루트)
```

### 3.6 Step 2 구현 및 추가 전략 (실측 기반)

Step 2는 `notebooks/`의 스크립트로 구현했다: `legal_common.py`(공용) → `02_eda.py` →
`02_chunking.py` → `02_concepts_conflicts.py` → `02_conditions.py`. 핵심 원칙은
**"엣지는 데이터에서 도출, QA는 구성에 절대 사용하지 않고 사후 recall 검증에만 사용"**(순환검증 회피).

**(1) 문서 레지스트리 + 타입드 참조 해소**

참조를 단순 추출하지 않고 `{target_doc, target_law, status}`로 분류한다:
`internal`(자기 문서) 4,814 / `cross_corpus`(코퍼스 내 타 문서) 2,011 / `external`(코퍼스 밖) 774.
시행령의 맨 "법 제N조"는 레지스트리로 상위법에 연결하되, 상위법이 코퍼스에 없으면(고보·근기)
external로 두고 **`Chunk.external_refs` 속성으로만 보존**(노드화 X → 그래프 오염 방지).
조특법↔조특법시행령만 상·하위법이 모두 있어 REFERENCES 엣지가 풍부하다.
**정합 검수(결정론적, `graph/audit_references.py`)**: 코퍼스 내 인용 **해소율 90.8%**(5,561/6,124),
해소 엣지 **항 도달성 85.6%**, 존재하지 않는 조문으로 가는 엣지 **0건**(그래프 오염 0). 미해소 9.2%는
대부분 시행령이 상위법 조문을 인용했으나 상위법이 코퍼스 밖이라 안전 격리. 문맥상 오연결은 소표본
육안 점검으로 보완(상세 `graph/reference_audit.md`).

**(2) 구조 인식 청킹 — EDA로 파라미터 확정**

항 단위가 기본 경계. EDA 실측(법령 median 120~165토큰)에 따라 **<300토큰 항은 같은 조 내
인접 항과 병합, >1500토큰 항은 호 단위 분할**, 각 청크에 상위 조 제목을 컨텍스트로 첨부.
결과 3,902청크(median 359토큰). 지침은 **페이지 앵커 + 토큰 상한(1000)**으로 분할, 표는 개별 청크.

**(3) CONFLICTS_WITH — 자동 후보 + 정준 조문 앵커링(QA 독립)**

순환검증을 원천 차단: ① 멀티문서 개념의 **정의·범위·산정 청크**를 자동 후보화(단순 언급 배제),
② 기준 수치 차이로 규칙 스코어 → **후보 70쌍**(`conflict_candidates.jsonl`). ③ 격리 LLM(gpt-4o)
이진판정을 시도했으나 **'같은 지표·다른 목적' 경계에서 정밀도가 불안정**(루브릭 표현에 따라
1↔25건 진동)함을 감사로 확인 → LLM은 후보 생성 보조로만 쓰고, **최종 엣지는 정준 조문에
앵커링한 전문가 확정 7건**(`02b_curate_conflicts.py`)으로 둔다. 각 엣지는 axis·legal_purpose·
note·verified를 보유. QA 정답은 미참조(사후 recall 점검만).
⚠️ 감사 교훈: 고보령 §12(우선지원대상기업 규모) ≠ §3①(피보험자 자격), 조특령은 '기간제'가
아니라 '계약 1년 미만'을 제외(§26의8②1호) — 용도·조문을 구분해 라벨링해야 함.

**(4) 지침 Condition — 타깃 LLM 구조화 추출**

지침은 반정형이라 구조 파싱 대신, 본문이 깨끗한 점을 활용해 핵심 섹션 청크(45개)를 LLM으로
엄격 JSON 추출 → **158 Condition**(type/numeric_criteria/applies_to/refs). 청년 15~34세,
주 30시간, 피보험자 5인, 12개월 등 QA 핵심 수치와 별첨·법령 참조를 캡처.
※ LLM이 추출한 '군필 가산 39세' 등 일부 지침 값은 **지침 원문 대조 미완료** → `verified=false`로
취급하고 단정 인용 금지(KG/메트릭 반영 전 검증 필요).

**(5) Concept 인벤토리**: 도메인 + EDA 빈도로 20종 확정(멀티문서 17종 = 충돌 모집단).

> **한계/Step3 인계**: ① DEFINES는 휴리스틱(산정식형 정의는 미탐, 충돌엣지로 보강). ② 동일개념
> 충돌만 자동 — "피보험자수↔상시근로자수" 같은 역할등가 cross-concept 충돌은 도메인 개념쌍을
> 1개 명시 입력해야 탐지(QA 누수 아님). ③ DELEGATES(구 OVERRIDES)는 레지스트리 위계 규칙으로 생성 —
> 법률→시행령 **위임관계**이며 '충돌 시 우선적용'과는 별개 개념(엣지 1건: 조특법→조특령).

## 4. Step 3 — Neo4j 기반 그래프 구축

### 4.1 왜 Neo4j인가

관계형 DB(MySQL, PostgreSQL)로도 조항 간 관계를 저장할 수 있지만, 법률 추론의 핵심인 **"A가 B를 참조하고, B가 C와 충돌하며, C의 정의에 따라 D가 적용된다"**와 같은 연쇄 관계(multi-hop traversal)를 표현하고 탐색하는 데는 그래프 DB가 압도적으로 유리하다.

Neo4j의 Cypher 쿼리는 이런 연쇄 관계를 직관적으로 표현할 수 있다:

```
(질문에서 언급된 개념)
  -[DEFINES]-> (개념을 정의하는 청크 A)
  -[REFERENCES]-> (A가 인용하는 청크 B)
  -[CONFLICTS_WITH]-> (B와 충돌하는 청크 C)
```

이 경로 탐색이 **hop**이고, 이 경로를 통해 검색된 청크 집합이 LLM에게 전달되는 컨텍스트이다.

### 4.2 그래프 구축 순서

1. **Document 노드 생성**: 5개 문서에 대한 최상위 노드
2. **Article 노드 생성**: 각 문서의 조(Article) 단위 노드
3. **Chunk 노드 생성**: all_chunks.jsonl에서 각 청크를 노드로 적재
4. **CONTAINS 엣지 생성**: Document → Article → Chunk 위계 연결
5. **NEXT_PARAGRAPH 엣지 생성**: 같은 조 내 인접 항 순서 연결
6. **REFERENCES 엣지 생성**: 청크 메타데이터의 references 필드를 파싱하여 대상 청크와 연결
7. **Concept 노드 + DEFINES/MENTIONS 엣지 생성**: 핵심 개념 노드를 만들고, 각 청크와 연결
8. **CONFLICTS_WITH 엣지 생성**: EDA에서 식별된 충돌 쌍을 수동+자동으로 라벨링

### 4.3 벡터 인덱스 — 하이브리드 검색의 핵심

그래프 탐색만으로는 "의미적으로 유사한" 청크를 찾기 어렵다. 예를 들어 사용자가 "해고할 때 세금 문제"라고 물으면, "해고"라는 단어가 직접 등장하지 않는 "상시근로자 수 감소 시 추징" 청크를 찾아야 한다.

이를 위해 **각 Chunk 노드에 텍스트 임베딩 벡터를 속성으로 저장**하고, Neo4j의 벡터 인덱스를 활용하여 의미 검색(semantic search)을 지원한다. 이렇게 하면:

- **1단계 (벡터 검색)**: 질문과 의미적으로 가까운 시작점(seed) 청크를 찾고
- **2단계 (그래프 탐색)**: 시작점에서 REFERENCES, CONFLICTS_WITH 엣지를 따라 연관 청크를 확장

이 **"벡터 검색 → 그래프 확장"** 2단계 검색이 GraphRAG의 핵심 메커니즘이다.

### 4.4 이 단계의 최종 결과물

```
graph/
```

### 4.5 Step 3 구현 및 추가 전략 (가이드 보강)

**(1) 임베딩과 그래프 적재의 분리 — 비용·재현성**

임베딩 생성(`build_embeddings.py`)을 DB 적재와 **분리**해 디스크에 캐시(`.npy`)한다. 임베딩은
비싸고 변하지 않으므로 한 번 만들어 재사용하고, Neo4j 재구축 시 재호출하지 않는다. 임베딩 입력은
**"상위 조 제목(context_header) + 본문"** — 맥락을 포함해 검색 정확도를 높인다(text-embedding-3-small, 1536d).

**(2) 오프라인 그래프 빌드 — DB 없이 모델 검증 + 폴백**

가이드 원안에 없던 단계. Neo4j와 **동일한 노드/엣지 정의**로 메모리 그래프를 만들어 ① 노드/엣지 수
정합성, ② 고립 청크·끊긴 참조 점검, ③ 멀티홉 동작(개념→정의청크→충돌상대)을 **API/DB 없이 즉시
검증**한다. 결과는 `offline_graph.pkl`로 직렬화해 Neo4j 미가동 시 Step 4의 폴백 저장소로도 쓴다.

**(3) 멱등 적재 + 타입드 엣지 충실 반영**

`build_graph.py`는 전부 `MERGE` + 유니크 제약으로 **재실행 안전**. REFERENCES는 **Chunk→Article**로
적재(참조는 조 단위가 자연스러움 → 이후 `CONTAINS`로 항 확장)하고 `status`(internal/cross_corpus)를
엣지 속성으로 보존. external은 엣지 대신 속성. DELEGATES는 레지스트리 위계(조특법→조특령 위임)로 생성.
Condition은 `HAS_CONDITION`으로 Document·출처 Chunk 양쪽에 연결(요건 추적성).

**(4) 벡터 인덱스 = 하이브리드 검색의 진입점**

Neo4j 네이티브 **벡터 인덱스**(cosine, 1536d)를 Chunk.embedding에 생성. Step 4 검색은
"질문 임베딩 → 벡터 top-k 진입점 → REFERENCES/CONFLICTS_WITH 엣지 확장"의 2단계로 동작한다.

**(5) 로컬 Neo4j 표준화**: `docker-compose.yml`(neo4j 5.26 + APOC)로 재현 가능한 로컬 인스턴스를
제공. 접속정보는 `.env`의 `NEO4J_URI/USER/PASSWORD`로 주입.

## 5. Step 4 — LangGraph 기반 에이전트 추론 시스템

### 5.1 전체 파이프라인 흐름

```
사용자 질문
    │
    ▼
┌──────────┐
│  Router  │  ← 질문 유형 분류 (단순 조회 / 다중 문서 추론 / 충돌 분석)
└────┬─────┘
     │
     ▼
┌──────────────┐
│  Retriever   │  ← 벡터 검색 → 그래프 확장 → 컨텍스트 수집
└────┬─────────┘
     │
     ▼
┌──────────────┐
│  Reasoner    │  ← 수집된 컨텍스트 기반 다중 문서 추론
└────┬─────────┘
     │
     ▼
┌──────────────┐     ┌─────────┐
│  Verifier    │────→│ REJECT  │──→ Retriever로 회귀 (최대 N회)
│ (검증 노드)   │     │ (리젝)   │
└────┬─────────┘     └─────────┘
     │ ACCEPT
     ▼
┌──────────────┐
│   Output     │  ← 구조화된 응답 + 근거 추적 정보 출력
└──────────────┘
```

### 5.2 Router 노드 — 질문 유형 분류

Router의 역할은 **질문의 복잡도와 필요한 추론 깊이를 판단**하여 후속 노드의 전략을 결정하는 것이다.

**분류 기준:**

| 유형 | 판단 기준 | Retriever 전략 |
|------|----------|---------------|
| **단순 조회 (hop 1)** | 단일 법률/조항에 대한 직접적 질문 | 벡터 검색으로 top-k 청크 반환, 그래프 확장 최소화 |
| **다중 문서 추론 (hop 2-3)** | "~와 ~를 비교", "~에 따르면 ~도 적용되는지" 등 교차 참조 필요 | 벡터 검색 후 REFERENCES 엣지 2-3홉 확장 |
| **충돌 분석** | "동시에 적용 가능한지", "어떤 차이가 있는지" 등 법령 간 비교 | Concept 노드 경유, CONFLICTS_WITH 엣지 우선 탐색 |

**Router가 하지 말아야 할 것**: 질문에 직접 답하는 것. Router는 오직 **분류 레이블과 탐색 전략 파라미터**만 출력한다. 답변 생성은 Reasoner의 몫이다.

### 5.3 Retriever 노드 — 그래프 기반 컨텍스트 수집

Retriever는 이 시스템의 **GraphRAG 핵심이 구현되는 곳**이다. 일반 RAG의 "검색"과 결정적으로 다른 점:

**일반 RAG의 검색:**
```
질문 임베딩 → 벡터 유사도 top-k → 청크 목록 반환 (끝)
```

**GraphRAG의 검색:**
```
질문 임베딩 → 벡터 유사도로 시드(seed) 청크 선정
    → 시드 청크에서 REFERENCES 엣지를 따라 인용된 조항 탐색
    → 시드 청크에서 DEFINES 엣지를 따라 관련 개념(Concept) 노드 탐색
    → Concept 노드에서 다른 문서의 DEFINES/MENTIONS 청크로 확장
    → CONFLICTS_WITH 엣지가 있으면 충돌 청크도 수집
    → 수집된 모든 청크를 관련도 순으로 정렬하여 반환
```

**이 차이가 왜 중요한가**(검증된 Q2 — 근기법 vs 세법 상시근로자 산정식 충돌):
- 일반 RAG는 "상시근로자 산정"으로 벡터 검색하면 한쪽 공식(예: 근기령 §7의2① "연인원÷가동일수")만
  가져와, 노무사가 낸 "상시 4.8명"을 통합고용세액공제 신청서에 **그대로 써도 되는 줄 안다.**
- GraphRAG는 그 청크에서 Concept("상시근로자") → **CONFLICTS_WITH** 엣지를 따라 조특령 §26의8⑥
  (월말 인원 평균 + 1년미만·월60h미만·임원·친족 제외)까지 수집해, "두 공식이 달라 **수치 전용 불가**"라는
  충돌 경고를 함께 낸다. (근기↔조특는 직접 참조가 없어 **Concept 노드가 유일한 다리** — §4.3 EDA와 일치)

**Retriever의 핵심 파라미터:**

| 파라미터 | 의미 | Router에서 결정 |
|---------|------|---------------|
| `seed_top_k` | 벡터 검색으로 가져올 시드 청크 수 | 단순 조회: 3, 다중 추론: 5 |
| `max_hops` | 그래프 확장 최대 깊이 | hop 1: 1, hop 2-3: 2~3 |
| `edge_priority` | 우선 탐색할 엣지 타입 | 충돌 분석: CONFLICTS_WITH 우선 |
| `max_context_chunks` | 최종 컨텍스트로 전달할 최대 청크 수 | 10~15 (LLM 컨텍스트 윈도우 고려) |

### 5.4 Reasoner 노드 — 다중 문서 추론

Reasoner는 Retriever가 수집한 컨텍스트 청크들을 받아 **최종 답변을 생성**하는 노드이다.

**Reasoner의 시스템 프롬프트 설계 방향:**

1. **역할 정의**: "당신은 법률·노무·세무 전문 컨설턴트입니다. 반드시 제공된 컨텍스트 내의 법문만을 근거로 답변하십시오."
2. **근거 명시 강제**: "답변의 모든 주장에 대해 [출처: 조특법 제29조의8 제6항] 형태로 근거 조항을 명시하십시오."
3. **충돌 감지 지시**: "동일한 사안에 대해 서로 다른 법률/지침이 다른 기준을 적용하는 경우, 이를 명시적으로 '충돌'로 표기하고 각각의 기준을 병렬적으로 서술하십시오."
4. **추론 과정 노출**: "답변에 도달하기까지의 논리적 단계를 번호를 매겨 서술하십시오."

**Reasoner가 하지 말아야 할 것**: 컨텍스트에 없는 법률 지식을 사전학습 지식(parametric memory)에서 끌어오는 것. 이것이 환각(hallucination)이며, Verifier에서 걸러야 하는 대상이다.

### 5.5 Verifier 노드 — 환각 검증 + 리젝 판단

Verifier는 Reasoner의 출력을 **자체 검증**하여 ACCEPT 또는 REJECT를 판단하는 노드이다. 이 노드가 이 시스템의 **신뢰성 보장 장치**이다.

**검증 기준 (체크리스트):**

| 검증 항목 | 판단 기준 | REJECT 조건 |
|----------|----------|-----------|
| **근거 존재성** | 답변에 인용된 모든 조항이 Retriever가 수집한 컨텍스트에 실제로 존재하는가? | 컨텍스트에 없는 조항을 인용한 경우 |
| **수치 정확성** | 답변에 언급된 수치(연령, 기간, 금액 등)가 컨텍스트의 원문과 일치하는가? | "2년" vs "3년" 같은 수치 불일치 |
| **논리 일관성** | 답변의 결론이 인용한 근거로부터 논리적으로 도출되는가? | 근거에서 "월 60h 미만 제외"인데 결론이 "55h도 포함" |
| **충돌 누락** | 관련 CONFLICTS_WITH 엣지가 존재하는데 답변에서 충돌을 언급하지 않은 경우 | 충돌 존재하나 미언급 |

**리젝 시 회귀 로직:**

REJECT된 경우, Verifier는 **리젝 사유를 구체적으로 기술**하여 GraphState에 기록한다. 이 리젝 사유는:

1. Retriever에게 **추가 탐색 지시**로 전달된다 (예: "제127조 제3항의 원문이 컨텍스트에 누락됨 → 해당 청크를 추가 검색하라")
2. Reasoner에게 **수정 지시**로 전달된다 (예: "2년이 아니라 3년이 원문 기준임 → 수치를 수정하라")

**무한 루프 방지**: GraphState에 `retry_count`를 두어 **최대 3회까지만 회귀**를 허용한다. 3회 초과 시 현재까지의 최선 답변 + "검증 미완료 항목" 경고를 함께 출력한다.

### 5.6 Output 노드 — 구조화된 응답 생성

최종 출력은 단순한 텍스트 답변이 아니라, **추론 과정의 추적 가능성(traceability)을 보장하는 구조화된 형식**이어야 한다.

**출력 포맷 구성:**

```
📋 답변:
  근기법용 '상시 4.8명'을 통합고용세액공제 신청서에 그대로 쓸 수 없습니다.
  두 제도의 상시근로자 산정식이 달라 별도 재산정이 필요합니다.

📎 근거 추적:
  1. [근로기준법시행령 §7의2①] — "산정기간(1개월) 연인원 ÷ 가동일수"
  2. [조세특례제한법시행령 §26의8⑥] — "매월 말일 인원 평균 + 1년미만·월60h미만·임원·친족 제외"

🔀 탐색 경로:
  시드: "상시근로자 산정" → [근기령_제7조의2_제1-2항]
    → DEFINES → Concept("상시근로자")
    → CONFLICTS_WITH → [조특령_제26조의8_제5-8항]

⚠️ 충돌 감지:
  - 상시근로자 산정식: 근기령(연인원÷가동일수) ↔ 조특령(월말 평균+제외) → 동일 인력도 결과가 달라
    수치 그대로 전용 불가
```

이 출력 구조가 프로젝트의 핵심 주장인 **"GraphRAG는 더 신뢰할 수 있는 답을 낸다"**를 증명하는 근거가 된다. 일반 RAG에서는 "탐색 경로"와 "충돌 감지"를 제공할 수 없다.

### 5.7 이 단계의 최종 결과물 (실제 구현 산출물)

처음 스캐폴딩은 `nodes/`·`prompts/` 폴더 분리를 제안했으나, 노드 5종과 프롬프트 3종이
서로 상태를 공유하고 분량이 작아 **단일 모듈로 통합**하는 편이 가독성·실행이 좋았다.
(빈 스텁 `nodes/`·`prompts/` 패키지는 제거했다.)

```
agent/
├── graph_state.py     # AgentState(TypedDict) + add_trace() — 상태 추적
├── retrieval.py       # Retriever: 벡터 시드 + 개념앵커/브리지/충돌/참조 확장 + 재랭킹
├── prompts.py         # ROUTER/REASONER/VERIFIER 시스템 프롬프트
├── nodes.py           # router/retriever/reasoner/verifier/output 노드 + 결정론 가드
├── app.py             # LangGraph 조립(검증 실패 시 retriever 회귀) + 실행 진입점
├── run_eval.py        # gold_set 평가 + 베이스라인 대비 지표 + 리포트 생성
└── eval_report.md     # 평가 결과 리포트(자동 생성)
eval/
└── gold_set.jsonl     # 평가셋(충돌 4 + distractor 2, gold 조항 정규화 토큰 포함)
```

### 5.8 구현 단계에서 추가한 최적화 전략 (가이드 기본 설계에 더함)

실측(§5.9)으로 검증한, 가이드 원안에 없던 핵심 전략들이다.

**(1) 개념 구동 충돌 앵커 (Concept-driven conflict anchoring) — 가장 큰 기여.**
벡터 검색만으로는 정준 충돌 조문(예: 단시간 → 고보령 §3 ↔ 조특령 §26의8)이 top-k에
들지 못한다(벡터 top-1이 부수 조문 조특령 §136의2 등에 끌림 — Step 3 점검에서 관찰).
해결: 질문에서 개념(상시근로자/단시간/피보험자 등)을 별칭 매칭으로 감지하고,
`CONFLICTS_WITH.concept`가 일치하는 **검증된 정준 충돌쌍을 양끝 모두 컨텍스트에 직접 주입**한다.
이 앵커는 최상위 재랭킹 점수(0.95)를 받아 항상 컨텍스트 상단에 온다. → Q1/Q10/Q11의
Context Recall이 0 → 1.0으로 상승.

**(2) 재랭킹 점수 위계.** 시드(벡터점수) < 참조(0.55) < 개념브리지(0.58) < 시드충돌(0.80)
< **정준충돌 앵커(0.95)**. "검증된 충돌 > 의미 다리 > 단순 인용" 순으로 신뢰도를 반영.

**(3) route별 edge_priority(가이드 5.3 파라미터의 구체화).**
simple=`[concept]`, multihop=`[references, concept]`, conflict=`[conflict, concept, references]`.
**충돌 앵커는 simple 라우트에서 발동하지 않게 게이팅** → distractor(단순 조회) 과잉탐지 0.

**(4) Verifier 결정론 가드 + 인용 보정(#10 함정 종단 해소).**
- LLM Verifier는 **근거 존재성·수치 충실성만** 판정(문체로 REJECT 금지).
- 충돌 커버리지·과잉탐지는 `conflict_flagged` + 충돌엣지 유무로 **결정론적 판정**(LLM 변동성 제거).
- Reasoner가 항 번호를 다르게 적어도(`§7의2 제3항` vs 청크 `제1-2항`) **조문 단위로 정규화해
  실제 chunk_id로 보정**한다. gold ↔ chunk_id 표기 불일치로 인한 거짓 환각 판정을 없앤다.

**(5) 회귀(retry)는 폭만 확대.** 검증 실패 시 seed_top_k·컨텍스트 수만 늘리고
**edge_priority(route 의미)는 바꾸지 않는다** → simple distractor가 재시도로 충돌 탐색으로
변질되어 과잉탐지하는 것을 방지.

**(6) 상태 추적(observability).** 모든 노드가 `state["trace"]`에 입·출력 요약을 누적해
"어느 엣지로 무엇을 확장했고 왜 ACCEPT/REJECT인지"를 단계별로 재구성할 수 있다.

### 5.9 실측 결과 (GraphRAG vs Naive RAG)

`eval/gold_set.jsonl`(충돌 4 + distractor 2) · `agent/eval_report.md`.

| 지표 | GraphRAG | Naive(벡터 top-k) |
|------|----------|------------------|
| 평균 Context Recall | **1.00** | 0.42 (Δ **+0.58**) |
| Conflict Detection(과잉탐지 미발생 포함) | **1.00** | — (제공 불가) |

이 격차가 "그래프 확장이 근거 수집률을 끌어올린다"는 프로젝트 핵심 주장의 정량 증거다.

### 6.2 핵심 평가 지표

> 본 프로젝트의 주장은 **"더 똑똑한 답이 아니라 더 신뢰할 수 있는(근거를 안 놓치는) 답"**이므로, **1순위는 Answer Accuracy가 아니라 Context Recall**이다.

| 우선 | 지표 | 측정 대상 | GraphRAG 우위 증명 포인트 |
|------|------|----------|----------------------|
| ★★★ | **Context Recall** | gold_articles의 근거 청크를 Retriever가 실제 수집했는가 | **핵심** — 그래프 확장이 근거 수집률을 높이는가 |
| ★★★ | **Hop별 누락률** | Context Recall을 hop 1/2/3으로 분리 (누락률 = 1−Recall) | **대표 차트** — hop 깊을수록 Naive 급증·Graph 유지(간격 벌어짐) |
| ★★ | **Conflict Detection** | conflict 문항(12개)에서 양쪽 기준을 모두 서술+충돌 명시 | CONFLICTS_WITH 엣지의 가치 |
| ★★ | **Conflict 오탐률** | distractor 문항(6개)에서 충돌을 잘못 외치지 않는가 | 과잉탐지 안 함 = 진짜 신뢰성 |
| ★ | **Faithfulness** | 답변이 수집 컨텍스트에만 기반(환각 없음) | Verifier 노드 효과(ON/OFF 비교) |
| ★ | **Answer Accuracy** | 최종 답변이 gold_answer와 의미 일치(LLM-judge) | 보조 — LLM이 근거 없이 우연히 맞힐 수 있어 1순위 아님 |

**⚠️ 측정 전 필수 점검 2가지:**

1. **chunk_id ↔ gold_article 매핑.** gold는 사람용 문자열(`"조특령 §26의8②2호"`), KG 청크는 ID(`"조특법시행령_26조의8_2항"`). 둘을 잇는 정규화 테이블(`eval/article_map.json`)이 없으면 매칭이 전부 실패해 Recall이 0으로 찍힌다. **A(KG)와 B(eval)가 청크 ID 규칙을 사전 합의.** 매칭은 조 단위로 느슨하게 시작 → 필요 시 항/호까지 좁힘.

2. **7개 CONFLICTS_WITH 엣지 ↔ 12개 conflict 문항 커버리지.** 확정 충돌 엣지는 7건인데 conflict 문항은 12개다. **12개 중 어느 것이 7개 엣지로 커버되는지 매핑을 먼저 만들어야** Conflict Detection이 정확히 측정된다. 특히 **Q10(피보험자수↔상시근로자수)** 같은 cross-concept 충돌은 자동 탐지에서 누락되므로(§3.6 한계), 7개 엣지에 포함됐는지 확인. 미포함 충돌 문항은 "엣지 없음 → 측정 불가"로 별도 표기(0점 처리 금지).

### 6.3 비교 실험 설계 (Baseline vs GraphRAG)

| 구분 | Naive RAG (Baseline) | GraphRAG (Ours) |
|------|---------------------|----------------|
| 검색 | 벡터 유사도 top-k만 | 벡터 시드 → 그래프 확장(REFERENCES/CONFLICTS_WITH) |
| 청킹 | **동일 청크 사용**(고정길이 아님) | 동일 청크 — 검색 방식만 다르게 하여 변인 통제 |
| 추론 | 단일 LLM 호출 | Router → Retriever → Reasoner → Verifier |
| 출력 | 답변만 | 답변 + 근거 추적 + 탐색 경로 |

> **변인 통제 주의:** Baseline과 GraphRAG가 **같은 청크·같은 임베딩**을 쓰고 *검색 방식만* 다르게 해야, 차이가 "그래프 확장" 효과임을 입증할 수 있다. 청킹까지 다르게 하면 무엇이 효과인지 분리 불가.

**비교 산출물(발표 핵심 슬라이드 2장):**
- ① **Hop별 누락률 꺾은선** — Naive 급증 vs Graph 유지 → 핵심 주장 증명
- ② **Conflict 감지 vs 오탐 2×2** — 충돌은 잡고 distractor엔 안 속음 → 신뢰성 증명

**최소 발표 성립선:** Context Recall(hop별)만 나와도 핵심 주장은 증명된다. Conflict까지 나오면 충돌 슬라이드가 살고, Faithfulness/Accuracy는 시간 남으면 부록.

## 7. GraphRAG 핵심 개념 정리

### 7.1 GraphRAG란 무엇인가

GraphRAG는 **"지식 그래프(Knowledge Graph)를 검색 보강 생성(RAG)의 검색 단계에 통합하는 방법론"**이다.

일반 RAG는 문서를 청크로 나누고, 질문과 벡터 유사도가 높은 청크를 가져와 LLM에 전달하는 방식이다. 이 방식의 근본적 한계는:

1. **관계 정보 소실**: 청크 A가 청크 B를 참조한다는 사실이 벡터 공간에서는 표현되지 않는다
2. **다중 홉 추론 불가**: "A → B → C"를 따라가야 하는 질문에서, A만 검색하고 B, C를 놓친다
3. **충돌 감지 불가**: 같은 개념에 대한 서로 다른 정의가 있을 때, 둘 다 가져오지 못하거나 무분별하게 섞는다

GraphRAG는 이 한계를 **문서 간 관계를 명시적 그래프로 구조화**함으로써 해결한다.

### 7.2 일반 RAG와의 핵심 차이

```
[일반 RAG]
  질문 → 벡터 검색 → 유사 청크 top-k → LLM → 답변
  (각 청크는 독립적, 관계 정보 없음)

[GraphRAG]
  질문 → 벡터 검색 → 시드 청크 → 그래프 탐색으로 관련 청크 확장
       → 탐색 경로 자체가 "추론의 근거 사슬"
       → LLM은 경로 정보를 포함한 컨텍스트로 추론 → 답변
```

### 7.3 이 프로젝트에서 GraphRAG가 특히 효과적인 이유

법률 도메인은 GraphRAG의 이점이 가장 극명하게 드러나는 분야이다:

1. **법률은 본질적으로 그래프 구조**: 법률은 다른 법률을 참조하고("제127조에 따라"), 시행령은 법률의 위임을 받으며("대통령령으로 정하는"), 지침은 법률을 해석한다. 이 관계가 이미 문서 내에 명시되어 있으므로, 그래프로 변환하는 것이 자연스럽다.
2. **동일 개념의 다중 정의**: "상시근로자", "청년" 같은 개념이 법률마다 다르게 정의된다. Concept 노드를 두고 각 정의와 연결하면 이 차이를 체계적으로 관리할 수 있다.
3. **추적 가능성(traceability) 요구**: 법률 자문에서는 "왜 이런 결론인가?"에 대해 조항 단위의 근거를 제시해야 한다. 그래프의 탐색 경로가 곧 근거 사슬이 된다.

### 7.4 Community Detection과 Global Search (참고)

Microsoft의 GraphRAG 논문에서는 **커뮤니티 감지(community detection)**를 통해 그래프의 노드를 의미적 클러스터로 묶고, 각 클러스터에 대한 요약을 생성하여 **글로벌 검색(global search)**에 활용하는 방식을 제안한다.

이 프로젝트에서는 문서 규모가 5개로 작기 때문에 커뮤니티 감지의 효과가 제한적이지만, 확장 시(예: 수십 개 법률을 다루는 경우) 고려할 수 있다. 현 단계에서는 **로컬 검색(local search)**에 집중한다 — 즉, 질문에서 시작하여 관련 노드를 점진적으로 확장하는 방식이다.

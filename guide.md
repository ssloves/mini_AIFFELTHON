## 2. Step 1 — 데이터 정제 (PDF → Markdown)

### 2.1 왜 마크다운인가

PDF는 **"보이는 대로"** 저장하는 포맷이다. 텍스트의 논리적 구조(제목, 조항, 항, 호의 위계)가 아니라 **좌표와 폰트 크기**로 레이아웃을 표현한다. 이 때문에:

- 같은 "제29조의8"이라는 조항이 PDF에서는 단순히 "특정 좌표의 텍스트 블록"일 뿐이고, 그 아래 제1항~제8항이 어디서 시작하고 끝나는지가 기계적으로 명확하지 않다.
- 표(table)는 좌표 기반 텍스트 배치로 표현되어, 행/열 구조가 추출 과정에서 깨지기 쉽다.
- 머리글/바닥글(페이지 번호, "법제처 국가법령정보센터" 등)이 본문과 섞인다.

**마크다운으로 변환하는 이유는 다음 단계(청킹·그래프 구축)에서 문서의 논리적 위계를 활용하기 위해서다.** 마크다운의 `#`, `##`, `###` 헤딩이 법률 문서의 조-항-호 구조와 자연스럽게 대응되며, 이 위계 정보가 곧 그래프의 CONTAINS 엣지가 된다.

### 2.2 마크다운 구조 설계 원칙

법률 문서의 구조적 특성에 따라 마크다운을 **다음과 같은 위계**로 정리한다:

```
# [문서 제목] ← 법률명/지침명 (예: 조세특례제한법)
## [편/장 제목] ← 있는 경우만 (예: 제3장 고용지원)
### [조 제목] ← 핵심 단위 (예: 제29조의8 통합고용세액공제)
#### 제N항 ← 조 아래의 각 항
- 제N호 ← 항 아래의 각 호
  - 가. 목 ← 호 아래의 각 목
```

**이렇게 설계하는 이유:**

1. **조(Article)가 그래프의 핵심 노드 단위**가 된다. 법률 해석에서 "제29조의8에 따르면..."이라고 인용하는 것처럼, 조가 법률 추론의 기본 참조 단위이다.
2. **항(Paragraph)이 청킹의 자연스러운 경계**가 된다. 하나의 조 안에서도 항마다 규율하는 내용이 다르기 때문에 (예: 제6항은 사후관리, 제8항은 위임 조항) 항 단위로 청킹해야 검색 정밀도가 올라간다.
3. **호(Subparagraph)와 목(Item)은 항에 종속**되므로, 항 청크 안에 포함시킨다. 호만 단독으로 떼어내면 맥락이 사라진다.

### 2.3 문서 유형별 정제 주의사항

| 문서 유형 | 핵심 주의점 |
|----------|-----------|
| **법률/시행령** (조특법, 고용보험법시행령, 근로기준법시행령) | 조-항-호-목 위계를 정확히 파싱. 부칙, 별표는 별도 섹션으로 분리. 개정일자 (`<개정 2024. 12. 31.>`) 메타데이터로 보존 |
| **행정지침** (장려금지침) | 장-절-항 대신 "유형Ⅰ/Ⅱ", "장", "절" 구조. 별첨(별첨1~5)은 본문과 별도로 처리하되, 본문에서 "☞ [별첨1]"로 참조하는 관계를 메타데이터로 기록 |

### 2.4 정제 시 보존해야 할 메타데이터

마크다운 변환 시 단순 텍스트만 남기면 안 된다. **다음 메타데이터를 YAML frontmatter 또는 인라인 태그로 보존**해야 그래프 구축에서 활용할 수 있다:

- **문서 식별자**: 법률명, 법률 번호, 시행일
- **조항 식별자**: 조 번호, 항 번호 (예: `article: 29조의8`, `paragraph: 6`)
- **개정 이력**: 최종 개정일 (시간적 유효성 판단에 필요)
- **상호 참조**: 본문 내에서 다른 조항을 인용하는 부분 (예: "제127조에 따라", "동법 시행령 제26조의8") — 이것이 그래프의 REFERENCES 엣지의 원천이 된다

### 2.5 이 단계의 최종 결과물

```
data/processed/
├── 장려금지침.md           # ~200페이지 분량의 구조화된 마크다운
├── 고용보험법시행령.md      # 조-항 구조
├── 근로기준법시행령.md      # 조-항 구조
├── 조특법.md               # 조-항-호 구조
└── 조특법시행령.md          # 조-항-호 구조
```

## 3. Step 2 — EDA 및 그래프 스키마 설계

### 3.1 EDA의 목적

EDA(탐색적 데이터 분석)의 목적은 단순히 "데이터를 살펴보는 것"이 아니다. **그래프를 어떤 입도(granularity)로 구축할지 결정**하기 위한 분석이다. 구체적으로:

1. **청킹 단위 결정**: 조 단위? 항 단위? 문장 단위? — 각 단위의 평균 토큰 수와 의미적 자족성(self-containedness)을 분석
2. **상호 참조 패턴 파악**: 어떤 조항이 어떤 조항을 참조하는지 빈도와 패턴 분석 — 이것이 REFERENCES 엣지의 밀도를 결정
3. **핵심 엔티티 추출**: 법률 문서에서 반복적으로 등장하는 핵심 개념(청년, 상시근로자, 세액공제, 피보험자 등) 식별 — 이것이 Concept 노드가 된다
4. **문서 간 충돌 지점 식별**: 같은 용어(예: "상시근로자", "청년")가 문서마다 다르게 정의되는 지점 — 이것이 CONFLICTS_WITH 엣지의 후보가 된다

### 3.2 청킹 전략 — "법률 구조 인식 청킹"

일반적인 RAG에서는 고정 길이(예: 512토큰)로 텍스트를 자르거나, 문단 단위로 나누는 방식을 쓴다. **법률 문서에서는 이 방식이 치명적이다.** 이유:

- **고정 길이 청킹의 문제**: "제29조의8 ⑥ 제1항에 따른 세액공제를 적용받은 사업자가..."라는 문장이 512토큰 경계에서 잘리면, 뒤에 오는 "3년 이내에 ... 감소한 경우에는 ... 납부하여야 한다"라는 핵심 결론이 다른 청크로 가버린다.
- **문단 단위 청킹의 문제**: 법률은 하나의 항이 수백 단어에 달하는 단일 문장인 경우가 많다. 문단 구분이 의미 단위와 일치하지 않는다.

**권장: 법률 구조 인식 청킹(Structure-Aware Chunking)**

```
청킹 단위 결정 기준:
├── 법률/시행령 → "항(Paragraph)" 단위
│   ├── 항이 300토큰 미만이면 → 같은 조의 인접 항과 병합
│   ├── 항이 1500토큰 초과이면 → 호(Subparagraph) 단위로 분할
│   └── 각 청크에 상위 조(Article) 제목을 컨텍스트로 첨부
│
└── 행정지침(장려금지침) → "소절 + 요건 블록" 단위
    ├── 표(table)는 하나의 청크로 보존 (표를 쪼개면 의미 소실)
    ├── "요건 내용" 블록 (➀, ➁, ➂ 등)은 하나의 청크로 묶기
    └── 각 청크에 상위 장/절 제목을 컨텍스트로 첨부
```

**각 청크에 반드시 포함해야 할 필드:**

| 필드 | 설명 | 예시 |
|------|------|------|
| `chunk_id` | 고유 식별자 | `조특법_29조의8_6항` |
| `source_doc` | 원본 문서명 | `조세특례제한법` |
| `article` | 소속 조항 번호 | `제29조의8` |
| `paragraph` | 소속 항 번호 | `제6항` |
| `hierarchy_path` | 위계 경로 | `조특법 > 제3장 > 제29조의8 > 제6항` |
| `text` | 청크 본문 | (실제 법문) |
| `references` | 본문 내 타 조항 참조 목록 | `["제132조", "시행령 제26조의8"]` |
| `keywords` | 핵심 개념어 | `["세액공제", "상시근로자", "추징"]` |

### 3.3 그래프 스키마 설계 — 노드(Node) 정의

법률 GraphRAG에서 노드는 **5가지 유형**으로 설계한다:

| 노드 레이블 | 의미 | 속성 | 인스턴스 예시 |
|------------|------|------|------------|
| `Document` | 하나의 법률/지침 문서 | `name`, `law_number`, `effective_date` | "조세특례제한법" |
| `Article` | 법률의 "조" 단위 | `number`, `title`, `full_path` | "제29조의8(통합고용세액공제)" |
| `Chunk` | 실제 검색·추론 단위 (항/요건 블록) | `chunk_id`, `text`, `embedding`, `token_count` | "제29조의8 제6항" |
| `Concept` | 법률 전체에 걸쳐 등장하는 핵심 개념 | `name`, `definition_sources` | "상시근로자", "청년", "세액공제" |
| `Condition` | 요건/조건 블록 (장려금지침 특화) | `type`, `description`, `applies_to` | "❹ 5인 미만 기업 참여 조건" |

**왜 Concept 노드가 필요한가?**

"상시근로자"라는 용어는 조특법시행령 제26조의8, 고용보험법시행령, 근로기준법시행령에서 **각각 다르게 정의**된다. Concept 노드 없이 텍스트 검색만 하면 "상시근로자"를 검색했을 때 3개 문서의 청크가 뒤섞여 나오고, 어떤 정의가 어떤 맥락에서 적용되는지 구분할 수 없다. Concept 노드를 두면 "상시근로자" → 각 문서별 정의 청크로의 명시적 연결이 생기고, 정의 간 차이(충돌)도 엣지로 표현할 수 있다.

### 3.4 그래프 스키마 설계 — 엣지(Edge) 정의

| 엣지 타입 | 시작 → 끝 | 의미 | 생성 방법 |
|----------|----------|------|----------|
| `CONTAINS` | Document → Article → Chunk | 포함 관계 (위계 구조) | 마크다운 헤딩 파싱으로 자동 생성 |
| `REFERENCES` | Chunk → Chunk | 법문 내 상호 인용 ("제127조에 따라") | 정규식으로 "제N조" 패턴 추출 후 매핑 |
| `DEFINES` | Chunk → Concept | 이 청크가 해당 개념을 정의함 | 키워드 + LLM 보조 추출 |
| `MENTIONS` | Chunk → Concept | 이 청크가 해당 개념을 언급함 (정의 X) | 키워드 매칭 |
| `CONFLICTS_WITH` | Chunk ↔ Chunk | 동일 개념에 대해 다른 기준 적용 | EDA에서 식별 + 수동 라벨링 |
| `OVERRIDES` | Chunk → Chunk | 상위법-하위법, 특별법-일반법 관계 | 법률 위계 규칙 기반 |
| `NEXT_PARAGRAPH` | Chunk → Chunk | 같은 조 내 인접 항 순서 | 항 번호 순서로 자동 생성 |

**엣지 중 가장 중요한 것: REFERENCES와 CONFLICTS_WITH**

- **REFERENCES**: 법률은 "제127조에 따라", "동법 시행령 제26조의8에 따라" 같은 상호 인용이 극도로 빈번하다. 이 인용 관계를 엣지로 명시하면, 질문에 대해 **단일 조항만 검색하지 않고 인용 사슬을 따라 관련 조항을 연쇄 탐색**할 수 있다. 이것이 GraphRAG가 일반 RAG보다 multi-hop 추론에 강한 핵심 이유이다.
- **CONFLICTS_WITH**: QA셋에서 "충돌있음"으로 표기된 질문들이 이 엣지를 통해 탐지된다. 예를 들어 조특법시행령의 "상시근로자" 정의와 장려금지침의 "피보험자 수" 산정법이 다르다는 것을, 두 청크 사이의 CONFLICTS_WITH 엣지로 표현한다.

### 3.5 이 단계의 최종 결과물

```
data/
├── chunks/all_chunks.jsonl         # 모든 청크 (chunk_id, text, metadata, references, keywords)
└── graph_schema.md                 # 노드/엣지 타입 정의, 속성 명세, 인스턴스 수 예상치
```

## 4. Step 3 — Neo4j 기반 그래프 구축

### 4.1 왜 Neo4j인가

관계형 DB(MySQL, PostgreSQL)로도 조항 간 관계를 저장할 수 있지만, 법률 추론의 핵심인 **"A가 B를 참조하고, B가 C와 충돌하며, C의 정의에 따라 D가 적용된다"**와 같은 연쇄 관계(multi-hop traversal)를 표현하고 탐색하는 데는 그래프 DB가 압도적으로 유리하다.

Neo4j의 Cypher 쿼리는 이런 연쇄 관계를 직관적으로 표현할 수 있다:

```
(질문에서 언급된 개념)
  -[DEFINES]-> (개념을 정의하는 청크 A)
  -[REFERENCES]-> (A가 인용하는 청크 B)
  -[CONFLICTS_WITH]-> (B와 충돌하는 청크 C)
```

이 경로 탐색이 **hop**이고, 이 경로를 통해 검색된 청크 집합이 LLM에게 전달되는 컨텍스트이다.

### 4.2 그래프 구축 순서

1. **Document 노드 생성**: 5개 문서에 대한 최상위 노드
2. **Article 노드 생성**: 각 문서의 조(Article) 단위 노드
3. **Chunk 노드 생성**: all_chunks.jsonl에서 각 청크를 노드로 적재
4. **CONTAINS 엣지 생성**: Document → Article → Chunk 위계 연결
5. **NEXT_PARAGRAPH 엣지 생성**: 같은 조 내 인접 항 순서 연결
6. **REFERENCES 엣지 생성**: 청크 메타데이터의 references 필드를 파싱하여 대상 청크와 연결
7. **Concept 노드 + DEFINES/MENTIONS 엣지 생성**: 핵심 개념 노드를 만들고, 각 청크와 연결
8. **CONFLICTS_WITH 엣지 생성**: EDA에서 식별된 충돌 쌍을 수동+자동으로 라벨링

### 4.3 벡터 인덱스 — 하이브리드 검색의 핵심

그래프 탐색만으로는 "의미적으로 유사한" 청크를 찾기 어렵다. 예를 들어 사용자가 "해고할 때 세금 문제"라고 물으면, "해고"라는 단어가 직접 등장하지 않는 "상시근로자 수 감소 시 추징" 청크를 찾아야 한다.

이를 위해 **각 Chunk 노드에 텍스트 임베딩 벡터를 속성으로 저장**하고, Neo4j의 벡터 인덱스를 활용하여 의미 검색(semantic search)을 지원한다. 이렇게 하면:

- **1단계 (벡터 검색)**: 질문과 의미적으로 가까운 시작점(seed) 청크를 찾고
- **2단계 (그래프 탐색)**: 시작점에서 REFERENCES, CONFLICTS_WITH 엣지를 따라 연관 청크를 확장

이 **"벡터 검색 → 그래프 확장"** 2단계 검색이 GraphRAG의 핵심 메커니즘이다.

### 4.4 이 단계의 최종 결과물

```
graph/
├── build_graph.py              # 그래프 구축 전체 스크립트
├── populate_embeddings.py      # 임베딩 생성 및 Neo4j 벡터 인덱스 적재
└── neo4j_snapshot/             # 구축 완료된 그래프 DB 덤프 (재현 가능)
```

**구축 완료 시 점검 항목:**

- Document 노드: 5개
- Article 노드: 수십~수백 개 (문서별 조 개수)
- Chunk 노드: 수백 개 (항 단위 청크)
- Concept 노드: 수십 개 (핵심 법률 개념)
- REFERENCES 엣지: 수백 개 (조항 간 상호 인용)
- CONFLICTS_WITH 엣지: 수십 개 (문서 간 충돌 지점)
- 벡터 인덱스: 모든 Chunk 노드에 임베딩 벡터 적재 완료

## 5. Step 4 — LangGraph 기반 에이전트 추론 시스템

### 5.1 전체 파이프라인 흐름

```
사용자 질문
    │
    ▼
┌──────────┐
│  Router  │  ← 질문 유형 분류 (단순 조회 / 다중 문서 추론 / 충돌 분석)
└────┬─────┘
     │
     ▼
┌──────────────┐
│  Retriever   │  ← 벡터 검색 → 그래프 확장 → 컨텍스트 수집
└────┬─────────┘
     │
     ▼
┌──────────────┐
│  Reasoner    │  ← 수집된 컨텍스트 기반 다중 문서 추론
└────┬─────────┘
     │
     ▼
┌──────────────┐     ┌─────────┐
│  Verifier    │────→│ REJECT  │──→ Retriever로 회귀 (최대 N회)
│ (검증 노드)   │     │ (리젝)   │
└────┬─────────┘     └─────────┘
     │ ACCEPT
     ▼
┌──────────────┐
│   Output     │  ← 구조화된 응답 + 근거 추적 정보 출력
└──────────────┘
```

### 5.2 Router 노드 — 질문 유형 분류

Router의 역할은 **질문의 복잡도와 필요한 추론 깊이를 판단**하여 후속 노드의 전략을 결정하는 것이다.

**분류 기준:**

| 유형 | 판단 기준 | Retriever 전략 |
|------|----------|---------------|
| **단순 조회 (hop 1)** | 단일 법률/조항에 대한 직접적 질문 | 벡터 검색으로 top-k 청크 반환, 그래프 확장 최소화 |
| **다중 문서 추론 (hop 2-3)** | "~와 ~를 비교", "~에 따르면 ~도 적용되는지" 등 교차 참조 필요 | 벡터 검색 후 REFERENCES 엣지 2-3홉 확장 |
| **충돌 분석** | "동시에 적용 가능한지", "어떤 차이가 있는지" 등 법령 간 비교 | Concept 노드 경유, CONFLICTS_WITH 엣지 우선 탐색 |

**Router가 하지 말아야 할 것**: 질문에 직접 답하는 것. Router는 오직 **분류 레이블과 탐색 전략 파라미터**만 출력한다. 답변 생성은 Reasoner의 몫이다.

### 5.3 Retriever 노드 — 그래프 기반 컨텍스트 수집

Retriever는 이 시스템의 **GraphRAG 핵심이 구현되는 곳**이다. 일반 RAG의 "검색"과 결정적으로 다른 점:

**일반 RAG의 검색:**
```
질문 임베딩 → 벡터 유사도 top-k → 청크 목록 반환 (끝)
```

**GraphRAG의 검색:**
```
질문 임베딩 → 벡터 유사도로 시드(seed) 청크 선정
    → 시드 청크에서 REFERENCES 엣지를 따라 인용된 조항 탐색
    → 시드 청크에서 DEFINES 엣지를 따라 관련 개념(Concept) 노드 탐색
    → Concept 노드에서 다른 문서의 DEFINES/MENTIONS 청크로 확장
    → CONFLICTS_WITH 엣지가 있으면 충돌 청크도 수집
    → 수집된 모든 청크를 관련도 순으로 정렬하여 반환
```

**이 차이가 왜 중요한가**: QA셋의 Q1을 예로 들면, "청년창업기업 대표자 연령"이라는 질문에 대해:
- 일반 RAG는 "청년" + "연령"으로 벡터 검색하면 장려금지침의 **39세** 기준 청크만 가져올 수 있다.
- GraphRAG는 39세 청크에서 REFERENCES 엣지를 따라 "5인 미만 기업 참여 조건" 청크로 이동하고, 그곳에서 **34세** 기준을 추가로 수집한다. 이것이 있어야만 "39세 정의와 34세 실제 참여 조건의 이원화 구조"를 파악할 수 있다.

**Retriever의 핵심 파라미터:**

| 파라미터 | 의미 | Router에서 결정 |
|---------|------|---------------|
| `seed_top_k` | 벡터 검색으로 가져올 시드 청크 수 | 단순 조회: 3, 다중 추론: 5 |
| `max_hops` | 그래프 확장 최대 깊이 | hop 1: 1, hop 2-3: 2~3 |
| `edge_priority` | 우선 탐색할 엣지 타입 | 충돌 분석: CONFLICTS_WITH 우선 |
| `max_context_chunks` | 최종 컨텍스트로 전달할 최대 청크 수 | 10~15 (LLM 컨텍스트 윈도우 고려) |

### 5.4 Reasoner 노드 — 다중 문서 추론

Reasoner는 Retriever가 수집한 컨텍스트 청크들을 받아 **최종 답변을 생성**하는 노드이다.

**Reasoner의 시스템 프롬프트 설계 방향:**

1. **역할 정의**: "당신은 법률·노무·세무 전문 컨설턴트입니다. 반드시 제공된 컨텍스트 내의 법문만을 근거로 답변하십시오."
2. **근거 명시 강제**: "답변의 모든 주장에 대해 [출처: 조특법 제29조의8 제6항] 형태로 근거 조항을 명시하십시오."
3. **충돌 감지 지시**: "동일한 사안에 대해 서로 다른 법률/지침이 다른 기준을 적용하는 경우, 이를 명시적으로 '충돌'로 표기하고 각각의 기준을 병렬적으로 서술하십시오."
4. **추론 과정 노출**: "답변에 도달하기까지의 논리적 단계를 번호를 매겨 서술하십시오."

**Reasoner가 하지 말아야 할 것**: 컨텍스트에 없는 법률 지식을 사전학습 지식(parametric memory)에서 끌어오는 것. 이것이 환각(hallucination)이며, Verifier에서 걸러야 하는 대상이다.

### 5.5 Verifier 노드 — 환각 검증 + 리젝 판단

Verifier는 Reasoner의 출력을 **자체 검증**하여 ACCEPT 또는 REJECT를 판단하는 노드이다. 이 노드가 이 시스템의 **신뢰성 보장 장치**이다.

**검증 기준 (체크리스트):**

| 검증 항목 | 판단 기준 | REJECT 조건 |
|----------|----------|-----------|
| **근거 존재성** | 답변에 인용된 모든 조항이 Retriever가 수집한 컨텍스트에 실제로 존재하는가? | 컨텍스트에 없는 조항을 인용한 경우 |
| **수치 정확성** | 답변에 언급된 수치(연령, 기간, 금액 등)가 컨텍스트의 원문과 일치하는가? | "2년" vs "3년" 같은 수치 불일치 |
| **논리 일관성** | 답변의 결론이 인용한 근거로부터 논리적으로 도출되는가? | 근거에서 "34세 이하"인데 결론이 "36세도 가능" |
| **충돌 누락** | 관련 CONFLICTS_WITH 엣지가 존재하는데 답변에서 충돌을 언급하지 않은 경우 | 충돌 존재하나 미언급 |

**리젝 시 회귀 로직:**

REJECT된 경우, Verifier는 **리젝 사유를 구체적으로 기술**하여 GraphState에 기록한다. 이 리젝 사유는:

1. Retriever에게 **추가 탐색 지시**로 전달된다 (예: "제127조 제3항의 원문이 컨텍스트에 누락됨 → 해당 청크를 추가 검색하라")
2. Reasoner에게 **수정 지시**로 전달된다 (예: "2년이 아니라 3년이 원문 기준임 → 수치를 수정하라")

**무한 루프 방지**: GraphState에 `retry_count`를 두어 **최대 3회까지만 회귀**를 허용한다. 3회 초과 시 현재까지의 최선 답변 + "검증 미완료 항목" 경고를 함께 출력한다.

### 5.6 Output 노드 — 구조화된 응답 생성

최종 출력은 단순한 텍스트 답변이 아니라, **추론 과정의 추적 가능성(traceability)을 보장하는 구조화된 형식**이어야 한다.

**출력 포맷 구성:**

```
📋 답변:
  (최종 답변 텍스트)

📎 근거 추적:
  1. [조특법시행령 제3조] — "창업 당시 15세 이상 34세 이하"
  2. [장려금지침 p.13 ❹] — "5인 미만 기업 참여 조건: 만 15세 이상 34세 이하"
  3. ...

🔀 탐색 경로:
  시드: "청년창업기업 연령" → [장려금지침_2장_청년창업기업]
    → REFERENCES → [장려금지침_2장_5인미만조건]
    → CONFLICTS_WITH → [조특법시행령_3조_청년기준]

⚠️ 충돌 감지:
  - 장려금지침 ➍(39세)와 ❹(34세)의 이원화 구조 확인
```

이 출력 구조가 프로젝트의 핵심 주장인 **"GraphRAG는 더 신뢰할 수 있는 답을 낸다"**를 증명하는 근거가 된다. 일반 RAG에서는 "탐색 경로"와 "충돌 감지"를 제공할 수 없다.

### 5.7 이 단계의 최종 결과물

```
agent/
├── graph_state.py          # LangGraph 상태 스키마 (query, context, answer, retry_count 등)
├── nodes/
│   ├── router.py           # 질문 유형 분류
│   ├── retriever.py        # 벡터 검색 + 그래프 확장
│   ├── reasoner.py         # 다중 문서 추론
│   └── verifier.py         # 환각 검증 + ACCEPT/REJECT
├── prompts/
│   ├── router_system.txt
│   ├── reasoner_system.txt
│   └── verifier_system.txt
└── app.py                  # LangGraph 워크플로우 조립 + 실행 진입점
```

## 6. 평가 전략

### 6.1 평가 데이터: qa.md

`qa.md`에 정의된 10개 QA 항목을 사용한다. 각 항목에는 질문, 정답(gold answer), 정답 근거(gold source), hop 수, 충돌 유형이 라벨링되어 있다.

### 6.2 핵심 평가 지표

| 지표 | 측정 대상 | GraphRAG 우위 증명 포인트 |
|------|----------|----------------------|
| **Answer Accuracy** | 최종 답변이 gold answer와 의미적으로 일치하는가 (LLM-as-a-Judge) | 기본 정확도 — 일반 RAG와 비교 |
| **Context Recall** | gold source에 명시된 근거 청크를 Retriever가 실제로 수집했는가 | **GraphRAG 핵심 지표** — 그래프 확장이 근거 수집률을 높이는지 |
| **Faithfulness** | 답변이 수집된 컨텍스트에만 기반하는가 (환각 없는가) | Verifier 노드의 효과 검증 |
| **Conflict Detection** | "충돌있음" 유형 질문에서 충돌을 정확히 감지하고 양쪽 기준을 모두 서술했는가 | CONFLICTS_WITH 엣지의 가치 증명 |
| **Hop Coverage** | hop 3 질문에서 3단계 이상의 문서 교차 참조를 성공적으로 수행했는가 | 그래프 탐색 깊이의 효과 검증 |

### 6.3 비교 실험 설계 (Baseline vs GraphRAG)

| 구분 | Naive RAG (Baseline) | GraphRAG (Ours) |
|------|---------------------|----------------|
| 검색 | 벡터 유사도 top-k만 사용 | 벡터 + 그래프 확장 |
| 청킹 | 고정 길이 또는 문단 단위 | 법률 구조 인식 청킹 |
| 추론 | 단일 LLM 호출 | Router → Retriever → Reasoner → Verifier |
| 출력 | 답변만 | 답변 + 근거 추적 + 탐색 경로 |

**비교 포인트**: 동일한 10개 QA에 대해 두 시스템의 Context Recall, Answer Accuracy, Conflict Detection 점수를 비교하여 GraphRAG의 우위를 정량적으로 입증한다.

## 7. GraphRAG 핵심 개념 정리

### 7.1 GraphRAG란 무엇인가

GraphRAG는 **"지식 그래프(Knowledge Graph)를 검색 보강 생성(RAG)의 검색 단계에 통합하는 방법론"**이다.

일반 RAG는 문서를 청크로 나누고, 질문과 벡터 유사도가 높은 청크를 가져와 LLM에 전달하는 방식이다. 이 방식의 근본적 한계는:

1. **관계 정보 소실**: 청크 A가 청크 B를 참조한다는 사실이 벡터 공간에서는 표현되지 않는다
2. **다중 홉 추론 불가**: "A → B → C"를 따라가야 하는 질문에서, A만 검색하고 B, C를 놓친다
3. **충돌 감지 불가**: 같은 개념에 대한 서로 다른 정의가 있을 때, 둘 다 가져오지 못하거나 무분별하게 섞는다

GraphRAG는 이 한계를 **문서 간 관계를 명시적 그래프로 구조화**함으로써 해결한다.

### 7.2 일반 RAG와의 핵심 차이

```
[일반 RAG]
  질문 → 벡터 검색 → 유사 청크 top-k → LLM → 답변
  (각 청크는 독립적, 관계 정보 없음)

[GraphRAG]
  질문 → 벡터 검색 → 시드 청크 → 그래프 탐색으로 관련 청크 확장
       → 탐색 경로 자체가 "추론의 근거 사슬"
       → LLM은 경로 정보를 포함한 컨텍스트로 추론 → 답변
```

### 7.3 이 프로젝트에서 GraphRAG가 특히 효과적인 이유

법률 도메인은 GraphRAG의 이점이 가장 극명하게 드러나는 분야이다:

1. **법률은 본질적으로 그래프 구조**: 법률은 다른 법률을 참조하고("제127조에 따라"), 시행령은 법률의 위임을 받으며("대통령령으로 정하는"), 지침은 법률을 해석한다. 이 관계가 이미 문서 내에 명시되어 있으므로, 그래프로 변환하는 것이 자연스럽다.
2. **동일 개념의 다중 정의**: "상시근로자", "청년" 같은 개념이 법률마다 다르게 정의된다. Concept 노드를 두고 각 정의와 연결하면 이 차이를 체계적으로 관리할 수 있다.
3. **추적 가능성(traceability) 요구**: 법률 자문에서는 "왜 이런 결론인가?"에 대해 조항 단위의 근거를 제시해야 한다. 그래프의 탐색 경로가 곧 근거 사슬이 된다.

### 7.4 Community Detection과 Global Search (참고)

Microsoft의 GraphRAG 논문에서는 **커뮤니티 감지(community detection)**를 통해 그래프의 노드를 의미적 클러스터로 묶고, 각 클러스터에 대한 요약을 생성하여 **글로벌 검색(global search)**에 활용하는 방식을 제안한다.

이 프로젝트에서는 문서 규모가 5개로 작기 때문에 커뮤니티 감지의 효과가 제한적이지만, 확장 시(예: 수십 개 법률을 다루는 경우) 고려할 수 있다. 현 단계에서는 **로컬 검색(local search)**에 집중한다 — 즉, 질문에서 시작하여 관련 노드를 점진적으로 확장하는 방식이다.
